# ST-09 · Ранковий звіт має бачити реальні вправи силової (не вигадувати)

**Тип:** покращення коректності · **Оцінка:** S–M · **Пріоритет:** високий (звіт
впевнено бреше про сьогоднішнє тренування) · **Залежності:** сусідить зі ST-05
(snapshot силової для `/plan`), ST-08 (валідація вправ)

## Проблема

Ранковий звіт 2026-07-13 (`report_logs` id 55) заявив: «🏋️ Сьогодні за планом
силова (W1)… сесія з **присіданнями/випадами/болгарськими**». Насправді сесія на
07-13 (`PlannedWorkout` id 368, активний план 9) — це **клон** Garmin-темплейта
931013083, і його вміст звіту невідомий.

Реконструкція справжнього `user_content`, що пішов у Claude (offline, лише читання
БД), підтвердила: у `plan_today` для силової пішло **лише**
`{"date":"2026-07-13","type":"strength","description":"Силова"}` — **жодної вправи**.
Модель добудувала «ноги» з `recent_activities`: останні силові користувача (07-09,
07-02, 06-25) — це Day 1 на ноги (болгарські присідання, випади, згинання ніг), і
модель прийняла їх за сьогоднішній план. Плюс прайм у `SYSTEM`: «Вправи на ноги
(присідання, випади…) перед бігом важать більше». Виходить впевнена галюцинація,
підкріплена історією, а не сьогоднішнім планом (реально це, найпевніше, Day 2 —
верх, тому «це не так»).

Дві незалежні дірки:

1. **`plan_today` ніколи не віддає вправи силової.** Білдер у
   `app/analysis/reports.py` (list-comprehension ~рядки 215-227) кладе тільки
   `date/type/dist_km/description/steps`. Для `type="strength"` перелік вправ
   (`strength_plan` для from-scratch, `strength_snapshot` для clone) не додається —
   тож навіть заповнений snapshot не дійшов би до звіту.

2. **Для clone-днів вправи не збережені локально — систематично, і причина
   підтверджена.** По всій БД **жоден** `PlannedWorkout.strength_snapshot` не містить
   реальних вправ: усі рядки або SQL `NULL` (плани до фічі), або текст `'null'` (плани
   8/9, де код snapshot виконався, але `snapshots.get(tid)` повернув `None`).

   **Корінь: `run_plan_generation` не автентифікує garth-клієнт перед strength-фетчами.**
   Функція будує весь контекст із БД (`plans.py:121-137` — `list_activities`,
   `read_history`, `weekly_run_volume`, `get_recent_extra`) — **жодного Garmin-виклику**.
   Далі strength-блок (`plans.py:169-182`) робить `client.fetch_workouts()` /
   `client.fetch_workout_full(tid)` — це **перші** Garmin-виклики у флові. Але
   `_UserGarthProvider.connectapi` (`providers.py:144`) **не викликає `self.login()`**, а
   саме `login()` (`providers.py:122`) робить `loads(garth_token)`, що завантажує токени.
   Ніхто інший теж не логіниться (`user_runtime` лише біндить ContextVar; `login()`
   лінивий — його зазвичай смикає `build_payload_cached`, якого тут немає). Тож
   `self._client` — порожній `garth.Client()` → `assert self.oauth1_token`
   (`garth/http.py:124`) кидається на **кожному** виклику. Обидва фетчі тихо деградують
   до `{}`/`[]` (`client.py:398,415`, ковтають `_error` без винятку) → `snapshots`
   порожній, `saved` порожній → рядок пишеться з `strength_snapshot=None` (→ JSON
   `null`) **і** з `description="Силова"` (справжнє ім'я темплейта втрачене — той самий
   симптом). Денний звіт цього не має, бо `build_payload_cached` логіниться першим.

   Підтверджено живим пробом (2026-07-13, read-only): той самий темплейт 931013083 без
   `login()` → `AssertionError`/`{}`; після `prov.login()` → `oauth1_token=True`,
   `workout_full → "Day 1"` з повним переліком вправ. `/plan`-в'юшка маскувала діру
   live-fetch-ом (`_strength_details` fallback), бо там `login()` уже стався в
   request-флові.

## Acceptance criteria

- [x] `plan_today` для `type="strength"` включає компактний перелік вправ, зібраний
      **з БД** у пріоритеті `strength_plan` (from-scratch) → `strength_snapshot`
      (clone): `{"name"?, "exercises":[{"category", "exercise"?, "reps"?}]}`. Обидва
      порожні → перелік не додаємо (лишаємо тільки `description`).
      → `reports._strength_exercises`, підключено в `plan_today` (цей PR).
- [x] `SYSTEM` (секція `plan_today`): «Описуючи силову, спирайся ТІЛЬКИ на
      `plan_today[].exercises`; якщо порожнє — не називай м'язову групу.
      `recent_activities` — ІСТОРІЯ силових, не сьогоднішній план.» (цей PR)
- [x] **Головний фікс: автентифікувати клієнт перед strength-фетчами.** Обрано
      **defense-in-depth**: `_UserGarthProvider.connectapi` (+ `username`/`display_name`)
      ліниво робить `self.login()` — тепер **будь-який** Garmin-виклик поза
      `build_payload_cached` автентифікується сам (login guard-ed/ідемпотентний, при
      валідному токені — просто `loads()`). Чинить і втрату імені
      (`description="Силова"`). (цей PR)
- [ ] Другорядно: **не ковтати тихо** порожній `fetch_workout_full`/`fetch_workouts` —
      залогувати гучний маркер (`PLAN strength snapshot empty tid=<id>`), щоб майбутні
      діри були видимі, а не мовчазні. _(не в цьому PR)_
- [ ] **Ідемпотентний бекфіл** наявних активних планів: заповнити `null`-снапшоти
      clone-днів одним проходом (читає темплейти live один раз, під `user_runtime` +
      `login()`; найближчий родич — CLI `backfill-*`). Врахувати JSON-null gotcha при
      виборі «порожніх» рядків.
- [ ] Тести: `plan_today` з clone-снапшотом → вправи в контексті; from-scratch
      `strength_plan` → вправи в контексті; обидва порожні → без вправ (промпт-гард
      покриває цей випадок).

## Підводні камені

- **JSON-null gotcha** (той самий, що в CLI-секції README): JSON-колонка зберігає
  Python `None` як JSON `null` (текст), тож `strength_snapshot is not null` ≠ «є дані».
  Фільтрувати «пусто» в Python (`json.loads(...) in (None, [], {})`), не через
  `.is_(None)`.
- Ранковий/денний звіт має лишатися **pure-DB** — жодного live-fetch темплейта в
  цьому шляху (Garmin-аутедж / MFA-ризик). Саме тому фіксимо через локальний snapshot,
  а не «дотягнути з Garmin у звіті».
- Категорії → людські назви: використати наявний `exercises.label()` /
  `exercise_names`, щоб у контекст ішли читані назви, а не Garmin-коди.

## Суміжні знахідки (з тієї ж реконструкції — можливо, окремі тікети)

- **День-фрейм у `previous_report`. [x] полагоджено в цьому PR.** Звіт 07-13 сказав
  «після **вчорашнього** тенісу+вело» — але теніс (62хв) і gravel-вело (120хв) були
  **07-11** (позавчора). Причина: вчорашній звіт (07-12) подається дослівно як
  `previous_report`, і там «вчора теніс+вело» було коректним *на 12-те*; модель
  перенесла фразу в 13-те без зсуву рамки. Реальне вчора (07-12) навантаження не мало
  (`auto_activities` — лише прогулянки + 11хв вело), а readiness впав до 51 через
  короткий сон/дефіцит REM (сам звіт 07-12 це визнав). → додано правило в `SYSTEM`
  (секція «ПОПЕРЕДНІЙ ЗВІТ»): відносні слова в `text` стосуються `previous_report.date`,
  не `today`; про дати навантаження судити лише за `recent_activities[].date` проти
  `today`.
- **`recovery_time_h` в хвилинах?** У `extra` за 07-12 `recovery_time_h=973`
  (=40 діб — абсурд як години; ~16 год як хвилини). Прайм-правило `SYSTEM`
  («`recovery_time_h > ~24` → обережність») спрацює хибно, якщо одиниця насправді
  хвилини. Перевірити мапінг у `service._daily_extra` проти сирого Garmin-поля.
