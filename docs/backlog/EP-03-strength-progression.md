# EP-03 · Прогресія силових

**Тип:** епік · **Оцінка:** L · **Пріоритет:** середній (коли силові стануть регулярними) ·
**Залежності:** немає (незалежний від EP-01/02)

## Сторя

> Як користувач із силовими днями в плані, я хочу, щоб ваги/сети/повтори
> прогресували від тижня до тижня (з deload-тижнями), а не повторювали одну
> й ту саму сесію весь план.

## Контекст

Зараз силові статичні: фіксована пара «день тижня → workout» на кожен тиждень
(clone Day 1/Day 2 або одна згенерована `strength_plan`-сесія, скопійована на всі
тижні). CLAUDE.md фіксує прогалину: «Not yet: AI progression on the exercises
(weights/sets over the plan)». Механіка пушу готова: `build_strength_workout`
будує нативний Garmin-DTO з будь-якого `strength_plan` (ваги в кг, -1.0 =
bodyweight), тож «різна сесія щотижня» — це лише інші дані на рядках.

**Скоуп**: прогресія ТІЛЬКИ для згенерованих сесій (`strength_plan`). Clone-дні
(Day 1/Day 2 шаблони користувача) не чіпаємо — шаблон належить користувачу,
мутувати його зміст від тижня до тижня означало б розсинхрон з тим, що він бачить
у своєму списку workout-ів.

## Acceptance criteria

- [ ] Згенерована силова відрізняється по тижнях: керований ріст (ваги +2.5–5 кг
      або +сет/повтори — за типом вправи), кожен 4-й тиждень — deload (-30–40% обсягу).
- [ ] Bodyweight-вправи прогресують повторами/сетами, не «вагою».
- [ ] Кожна тижнева сесія проходить `_sanitize_strength` (жодна галюцинована
      категорія не досягає годинника).
- [ ] `/plan`-акордеон показує конкретну сесію свого тижня (він уже читає
      `strength_plan` per-workout — має запрацювати без змін, перевірити).
- [ ] На годинник пушиться сесія відповідного тижня (rolling window бере правильний
      рядок — теж уже так, перевірити).
- [ ] Чат-едіт силової (`swap_exercise` / перегенерація дня) редагує ЛИШЕ той
      тиждень, не ламаючи прогресію інших.
- [ ] Назва workout-а несе тиждень: `🏋️ <name> · W3` (патерн уже є в clone-шляху).

## План по файлах

- `app/analysis/prompts.py` — `SYSTEM_STRENGTH_GEN`: замість однієї сесії просити
  масив по тижнях `{weeks: [StrengthSession, ...]}` із явними правилами прогресії
  та deload; на вхід — кількість тижнів плану.
- `app/garmin/schemas.py` — `StrengthProgression = {weeks: list[StrengthSession]}`
  (або просто валідувати список); `StrengthSession` без змін.
- `app/analysis/service.py` — `generate_strength_with_stats` (~:644): параметр
  `weeks: int`; валідація кожного елемента; fallback — якщо модель повернула одну
  сесію, розмножити її (деградація до поточної поведінки, не помилка).
  `run_plan_generation` (~:675): передавати кількість тижнів; дедуп однакових
  описів лишити (один опис → одна прогресія).
- `app/garmin/repository.py` — `add_strength_workouts`: для custom-описів класти
  на тиждень N сесію `weeks[N-1]` (замість однієї на всі); clone-гілка
  (`garmin_template_id`) без змін.
- `app/garmin/plan_sync.py` / `app/garmin/workout_export.py` — без змін по суті;
  у `build_strength_workout` перевірити, що ім'я включає тиждень.
- `app/routers/plan.py` — `_strength_details` вже читає per-workout
  `strength_plan`; перевірити рендер різних тижнів.
- `tests/` — розкладання прогресії по рядках; fallback однієї сесії; санітизація
  кожного тижня; чат-едіт тижня N не чіпає N+1.

## Підводні камені

- **Розмір відповіді**: 12-тижневий план × повна сесія — стежити за
  `max_tokens` виклику генерації силової (окремий від плану виклик — підняти ліміт).
- Прогресія «в один бік»: без факту виконання (Garmin не віддає нам виконані
  сети зараз) це відкрита петля — чесно зафіксувати в описі фічі; закриття петлі
  по силових — окрема майбутня сторя, якщо знайдеться endpoint.
- Вже згенеровані плани: рядки зі старою «однаковою» сесією лишаються валідними —
  міграція не потрібна, прогресія стосується нових генерацій.
