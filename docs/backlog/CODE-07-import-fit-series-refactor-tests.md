# CODE-07 · Розплутати `import_fit_series` + тести (найскладніша функція без тестів)

**Тип:** рефакторинг + тести · **Оцінка:** S–M · **Пріоритет:** низький (разовий
офлайн-інструмент), але це найризиковіший непокритий код, що пише в БД ·
**Залежності:** —

## Проблема

`export_import.import_fit_series` (`app/garmin/export_import.py:219`) — рекордсмен
проєкту за складністю: 96 рядків, cyclomatic 20, **вкладеність 8**, CRAP 420,
жодного теста. Вона в одному тілі: сканує FIT-файли експорту, будує індекс
session-start-time → файл, матчить до пробіжок за `beginTimestamp`, парсить
рекорди через `fitparse`, даунсемплить у `[{d, p, hr}]` і пише `series` в БД.
Помилка тут або тихо пропускає активності, або (гірше) пише криві серії, які
потім рендеряться графіками і їдуть у `/activity`-аналіз до Claude.

Сусіди `import_export`/`_build_day` покриті тестами (`tests/test_export_import.py`)
— цей шлях лишився сліпою зоною.

## Acceptance criteria

- [ ] Розбити на чисті функції: (1) скан FIT-каталогу → `{start_ts: path}`;
      (2) матчинг активності до FIT за `beginTimestamp` (з наявним допуском);
      (3) парсинг одного FIT → сирі точки; (4) даунсемпл до ~150 точок
      (якщо він дублює логіку live-шляху `fetch_activity_series` — витягти
      спільний хелпер, це та сама форма `[{d, p, hr}]`).
- [ ] Вкладеність ≤ 3; оркеструюча `import_fit_series` — тонкий цикл над
      хелперами. CLI-поведінка (`import-fit-series --email --path [--since]`)
      без змін, включно з «fills nulls only» (JSON-null фільтрується в Python —
      задокументована пастка).
- [ ] Тести на хелпери **без** справжніх FIT-файлів: `fitparse` мокається /
      фейкові record-об'єкти; окремо тест матчингу (збіг, розбіжність у кілька
      секунд, відсутність пари) і тест «series вже є → не перезаписувати».
- [ ] Поки поруч: `_build_day` (`:79`, вкладеність ок, але fan-out 16) не чіпати —
      він покритий і цілісний.

## Підводні камені

- FIT-імена файлів ≠ activity id (задокументовано в CLAUDE.md) — матчинг лише
  за session start time; тест на «два FIT з близьким стартом» обов'язковий.
- `fitparse` — опційна залежність цього шляху: імпорт всередині функції,
  щоб решта `export_import` працювала без нього. Не піднімати імпорт на рівень
  модуля.
- Це backfill-інструмент: ідемпотентність (повторний прогін нічого не міняє) —
  частина контракту, закріпити тестом.
