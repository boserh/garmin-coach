# ST-05 · Прев'ю згенерованої силової в setup-формі

**Тип:** покращення · **Оцінка:** M · **Пріоритет:** середній (філер) · **Залежності:** немає

## Сторя

> Як користувач, що обирає «🆕 інше…» (силова з опису) у формі створення плану,
> я хочу побачити згенеровану сесію (вправи/сети/ваги) ДО генерації всього плану —
> щоб не чекати повну генерацію заради силової, яка вийшла не такою.

## Контекст

Зараз free-text опис силової (`strength_desc_<slug>`) перетворюється на
`StrengthSession` лише всередині `run_plan_generation` (через
`generate_strength_with_stats`, дедуп однакових описів, `_sanitize_strength`).
Результат користувач бачить уже на готовому плані. Чат-шлях («додай силову на ноги»)
має підтвердження через ✅/❌ — форма ні.

## Acceptance criteria

- [ ] Біля текстового поля опису — кнопка «Прев'ю»: показує згенеровану сесію
      (назва, блоки, вправи з українськими назвами, sets×reps, ваги) без сабміту форми.
- [ ] Прев'ю рендериться тим самим виглядом, що акордеон силової на `/plan`.
- [ ] Підтверджене прев'ю НЕ генерується вдруге під час генерації плану
      (сесія кешується і передається в intake).
- [ ] Змінив опис після прев'ю → на генерації використовується актуальний опис
      (застаріле прев'ю інвалідується).
- [ ] Виклик прев'ю логується в `ReportLog` (kind="strength" або наявний kind
      генерації силової) — вартість видима.
- [ ] Без прев'ю форма працює як зараз (крок опційний).

## План по файлах

- `app/routers/plan.py` —
  - новий route `POST /plan/strength/preview`: приймає description, кличе
    `generate_strength_with_stats` + `repository._sanitize_strength`, повертає
    HTML-фрагмент (той самий формат, що будує `_strength_details` для
    `strength_plan`-днів);
  - `plan_create` (~:376): якщо у формі прийшла підтверджена згенерована сесія
    (hidden input з JSON + hash опису) — покласти в
    `intake["strength"]["custom_generated"][slug]`.
- `app/analysis/service.py` — `run_plan_generation` (~:675): для описів, що мають
  `custom_generated` зі збіжним hash — пропустити генерацію, взяти готову сесію
  (все одно прогнати через `_sanitize_strength` — не довіряти клієнтському JSON).
- `app/templates/plan.html` — кнопка «Прев'ю» + контейнер результату; мінімальний
  fetch() у стилі наявного inline-JS (прогресивне покращення: без JS форма
  працює по-старому); hidden inputs `strength_preview_<slug>` / hash.
- `tests/` — route прев'ю (mock генерації); тест пропуску повторної генерації
  при збіжному hash і НЕ-пропуску при зміненому описі.

## Підводні камені

- **Не довіряти клієнту**: сесія з hidden input повертається з браузера — обов'язкова
  повторна `_sanitize_strength` (валідація категорій проти Garmin-каталогу) на бекенді.
- Інвалідація: ключувати прев'ю hash-ем опису; розійшовся hash → генерувати заново.
- Прев'ю — платний виклик Claude: кнопка з явним лейблом, не автогенерація на input.
