# ST-01 · Ранковий звіт бачить сьогоднішнє тренування з плану

**Тип:** покращення · **Оцінка:** S · **Пріоритет:** найвищий · **Залежності:** немає

## Сторя

> Як бігун з активним тренувальним планом, я хочу, щоб ранковий звіт (і `/report`)
> коментував саме заплановану на сьогодні/завтра сесію з мого плану з огляду на
> HRV/readiness — щоб зранку знати, бігти за планом чи полегшити.

## Контекст

Зараз звіт бачить лише `planned_runs` з Garmin-календаря (те, що ми *читаємо*),
але не бачить сесій нашого власного `TrainingPlan`/`PlannedWorkout` (те, що ми
*прописуємо*). Якщо синк у календар вимкнено (`garmin_sync_enabled=False`) або
workout ще не в вікні пушу — звіт взагалі не знає, що сьогодні темпова.

## Acceptance criteria

- [ ] Якщо в активному плані є сесія на сьогодні або завтра, її type/dist/description/steps
      потрапляють у контекст `run_analysis` для kind `report`/`morning` (не `deep`).
- [ ] `SYSTEM`-промпт інструктує: оцінювати готовність саме до цієї сесії; за поганого
      HRV/низького readiness — пропонувати конкретне полегшення (коротше/повільніше/перенести).
- [ ] Немає активного плану / немає сесій у вікні → поведінка як зараз (без порожніх блоків).
- [ ] Dedup-cache key враховує план: зміна сесії (напр. після `/plan <edit>`) → новий звіт,
      а не кеш-хіт.
- [ ] Повторний `/report` того ж дня без змін плану → `CLAUDE CACHE HIT` (ключ стабільний).

## План по файлах

- `app/garmin/repository.py` — новий хелпер `upcoming_plan_workouts(session, user_id, days=2)`:
  `get_active_plan` + `list_workouts`, фільтр `status="planned"`, дати в `[today, today+days)`.
  Повертає компактні dict'и `{date, type, dist_km, description, steps?}` (steps — стисло,
  без зайвої вкладеності; це йде в платний контекст).
- `app/analysis/service.py` —
  - `run_analysis` (~:516): для `kind != "deep"` викликати хелпер і передати результат далі;
  - `analyze_with_stats` (~:172): новий параметр `plan_today: Optional[list]`, додати в
    `user_content`;
  - `_cache_key` (~:121): включити `plan_today` у хеш.
- `app/analysis/prompts.py` — доповнити `SYSTEM` правилом реакції на сьогоднішню сесію
  (та ж «proximity»-логіка, що вже є для погоди й темпу: детально — лише якщо сесія
  сьогодні/завтра).
- `tests/` — юніт на хелпер (вікно дат, лише `planned`, user-scoped); тест, що cache key
  змінюється при зміні плану і стабільний без змін.

## Підводні камені

- **Cache key** — головне: забути включити `plan_today` = звіт «не бачитиме» правок плану.
- Не плутати з `payload.planned_runs` (Garmin-календар) — це окреме джерело; у промпті
  варто пояснити різницю або віддавати перевагу нашому плану, коли він є.
- Бот/веб-роутери не чіпаються — все всередині `run_analysis`, тож і `/report.json`
  отримує це безкоштовно.
