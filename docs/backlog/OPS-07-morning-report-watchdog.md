# OPS-07 · Watchdog жорсткого збою ранкового звіту о deadline

**Тип:** надійність (BA-аналіз 2026-07-23, кат. B; пере-скоуплено після звірки з кодом) ·
**Оцінка:** S · **Пріоритет:** низький · **Поверхня:** бот · **Залежності:** сильно
перетинається з OPS-05 (Garmin-помилки) — розглядати разом, можливо злити

## Проблема

⚠️ **Уточнення після звірки з кодом**: первісне формулювання цього тікета
(«будь-який збій вікна → тиша») було **значною мірою хибним** — більшість шляхів уже
покриті, і головний кейс «годинник не синкнувся» теж:

- **Годинник не синкнувся (soft)**: `_deliver_morning` (`bot/jobs.py:181`) о deadline
  (`MORNING_DEADLINE_HOUR=12`) **усе одно шле звіт** зі stale-приміткою
  (`bot/jobs.py:188,194`, текст `_MORNING_STALE`: «⚠️ Дані за сьогодні ще не
  синканулись, звіт за останній доступний день»). Нотифікація про несинхронізований
  годинник **уже існує**.
- **`MFARequired`**: уже DM раз/добу (`_tick_for_user`).
- **Claude-помилка**: `delivery.build_report` у try/except (`bot/jobs.py:206`) — шле
  текст помилки, не тишу.
- **Garmin soft-помилки (5xx на ендпоінтах, логін ок)**: payload будується, today
  `has_data=False` → той самий stale-звіт о 12:00.
- **Telegram тимчасово впав**: тік ретраїться кожні 20 хв.

Лишається **одна вузька щілина**: **жорсткий збій логіну Garmin о deadline** —
протухлий токен без чистого `MFARequired`, Cloudflare-блок — коли
`build_payload_cached` кидає виняток **до** того, як зʼявиться payload, тож навіть
stale-примітка не спрацьовує, а MFA-DM не йде (це не `MFARequired`). Тоді день
справді минає без жодного сигналу. Цю саму щілину значною мірою покриває **OPS-05**
(там auth-fail шле DM одразу) — тому цей тікет має сенс лише як тонкий доповнювач до
OPS-05 або взагалі як його частина.

## User story

Як користувач я хочу отримати наприкінці ранкового вікна повідомлення, якщо звіт не
вдалося зібрати через жорсткий Garmin-збій (навіть stale-звіт не пішов), щоб не гадати
ввечері, чому тихо, і одразу знати, що робити (перевірити /settings, смикнути /report).

## Обсяг

**Входить:** у `_tick_for_user` — рахувати лише спроби, що впали **до** побудови
payload (тобто `build_payload_cached` кинув, не досягли `_deliver_morning`): лічильник
`morning_hard_fail:<date>` у `bot_state`; якщо `now` в останній годині вікна
(11:00–12:00 за таймзоною юзера, ST-14), guard `morning_sent_date` не стоїть, stale-звіт
теж не пішов і лічильник >0 → один DM «⚠️ Ранковий звіт не зібрався (жорсткий збій
Garmin, N спроб): <остання причина>. Перевір /settings або смикни /report», guard
`morning_hard_fail_notified:<date>`. Детерміноване — нуль LLM, нуль Garmin.

**Не входить:** сигнал про soft-кейси (stale-звіт їх уже покриває — watchdog про них
мовчить); watchdog для digest/plan_sync; дублювання OPS-05-логіки (якщо OPS-05
реалізовано першим і вже шле auth-fail DM — цей тікет закривається як покритий).

## Acceptance criteria

- [ ] DM іде **тільки** коли навіть stale-звіт не пішов (жорсткий pre-payload збій) —
      день зі звітом (звичайним АБО stale) → нуль DM.
- [ ] «Годинник не синкнувся» (пішов stale-звіт) НЕ вважається збоєм — тиша (це вже
      наявна поведінка, тест її фіксує).
- [ ] Рівно один DM в 11:00–12:00 локального часу юзера; причина — остання
      зафіксована (тип винятку).
- [ ] Якщо OPS-05 уже шле auth-fail DM для цього ж збою — не дублювати (спільний guard
      або залежність).
- [ ] Тести: hard-fail→DM раз, stale-звіт→тиша, звичайний звіт→тиша, ізоляція.

## Технічні дотики

- `bot/jobs.py::_tick_for_user` (лічильник pre-payload збоїв + перевірка хвоста вікна),
  константи вікна вже в модулі (`MORNING_START_HOUR`/`MORNING_DEADLINE_HOUR`).
- `app/garmin/repository/state.py` (get/set_state — наявні хелпери).
- Координація з OPS-05 (щоб один auth-fail не дав два DM).

## Вартість LLM / запити до Garmin

0 / 0 (повідомлення детерміноване; сам тік і так уже пробував фетчити).

## Ризики

- **Головний — надмірність**: після OPS-05 залишок цінності малий. Рекомендація —
  реалізовувати ПІСЛЯ OPS-05 і лише якщо auth-fail DM звідти не покриває «ще й stale
  не пішов»-нюанс; інакше закрити як дубль.
- Тік може не спрацювати рівно об 11:00+ (Pi перезавантажився) — вікно «остання
  година», не точна хвилина.
