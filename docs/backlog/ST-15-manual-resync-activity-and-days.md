# ST-15 · Ручний ресинк однієї активності та діапазону днів

**Тип:** покращення (BA-аналіз 2026-07-23, кат. A — керування даними) · **Оцінка:** M ·
**Пріоритет:** високий · **Поверхня:** обидва (веб — кнопка/форма, бот — команда) ·
**Залежності:** — (ST-16 і ST-19 будуються поверх цього)

## Проблема

Минулі дні назавжди віддаються з БД: `service.build_payload_cached` фетчить лише
дні, яких **немає** в `daily_metrics` (`to_fetch = [d for d in dates if not (d < today
and iso in cached)]`), а `repository.core.upsert_activity` оновлює активність лише
коли вона потрапила у вікно останніх N активностей чергового фетчу. Якщо активність
відредаговано в Garmin (обрізаний трек, виправлений тип), день записався криво, або
запис задвоївся — користувач не має жодної кнопки «підтягни ще раз»: лише CLI
`backfill-series` (який заповнює **тільки** null-серії) або пряма робота з БД.

## User story

Як користувач я хочу натиснути «Ресинк» на конкретній активності або задати діапазон
дат, щоб виправлені/недокачані дані з Garmin потрапили в БД без ssh до Pi.

## Обсяг

**Входить:**
- Веб: кнопка «🔄 Ресинк» на детейл-сторінці активності (`/me/activities/{id}`) —
  перетягує summary, series (для бігу/вело), splits і exercise sets цієї активності
  та перезаписує рядок.
- Веб: форма «Ресинк діапазону днів» (на `/settings` або `/me`): дата з/по,
  жорсткий кап 31 день — force-перефетч `daily_metrics` цих днів (upsert поверх).
- Бот: `/resync [YYYY-MM-DD [YYYY-MM-DD]]` (без аргументів = вчора+сьогодні).

**Не входить:** обхід дискового кешу series/sets (ST-16 — окремий прапорець зверху);
перегенерація Claude-аналізу (ST-19); автоматичне виявлення неповних днів (ST-18).

## Acceptance criteria

- [ ] Ресинк активності оновлює `dur_min`/`dist_km`/`avg_hr`/`series`/`exercises`
      наявного рядка (не створює дубль) і працює для активності старшої за вікно
      `activity_limit`.
- [ ] Ресинк діапазону перезаписує `daily_metrics` (включно з `extra`) для кожного
      дня діапазону; діапазон >31 дня відхиляється з людським повідомленням.
- [ ] Обидва шляхи йдуть під `service._user_fetch_locks` і через `client._api`
      (rate limiter `GARMIN_RPS` + 429-retry успадковуються безкоштовно).
- [ ] `MFARequired` під час ресинку → наявний флоу (409 + банер «заверши логін у
      /settings» на вебі; дружнє повідомлення в боті), не stack trace.
- [ ] Тести: upsert-поверх (без дубля), кап діапазону, чужа активність → 404.

## Технічні дотики

- `app/garmin/service.py`: нові `resync_days(session, user_id, dates)` (реюз
  `_fetch_days` + `repository.core.upsert_daily`) і `resync_activity(session, user_id,
  activity_db_id)` (фетч по `activity_id`: summary з `_activity_rows`-механіки або
  точковий виклик, series через `fetch_activity_series(sport=...)` з
  `multisport.sport_bucket`).
- `app/routers/me.py`: `POST /me/activities/{id}/resync`; форма діапазону + POST.
- `bot/handlers.py`: `resync` + реєстрація в `bot/main.py`.
- БД/міграції: немає (пише в наявні таблиці).

## Вартість LLM / запити до Garmin

LLM: 0 (свіжі дані змінюють dedup-ключ — наступний звіт буде платним рівно настільки,
наскільки дані справді нові; це коректно). Garmin: ~4–6 запитів на день діапазону,
~3–4 на активність; разова ручна дія, під наявним лімітером.

## Ризики

- Великий діапазон + 429 → кап 31 день, послідовний фетч, підсумок «N з M днів»
  замість мовчазного падіння.
- Ресинк перезаписує ручні правки полів? Ні — вручну редагованих полів немає;
  `subjective`/`analysis`/`step_match` не чіпати (перелічити явно у тесті).
