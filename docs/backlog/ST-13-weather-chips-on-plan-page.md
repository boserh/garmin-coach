# ST-13 · Прогноз погоди у веб-в'ю `/plan`

**Тип:** покращення (BA-аналіз 2026-07-21) · **Оцінка:** S · **Пріоритет:** низький
(видимість уже наявної логіки) · **Залежності:** EP-13 ✅
(`fetch_forecast_week`/`find_weather_conflicts`)

## Проблема

EP-13 двигає сесії через Telegram-пропозиції, але на сторінці плану погоди не
видно — користувач бачить «перенесено», та не бачить *чому*, і не може сам глянути
тиждень наперед перед ручним едитом.

## Acceptance criteria

- [ ] `GET /plan` (і read-only `/plan/{id}` — ні, лише активний план: минулим
      прогноз не потрібен): для сесій наступних 7 днів — компактні чіпи біля дати
      (🌡️ feels-max / 🌧️ prob / 💨 wind) з тижневого форкасту користувача.
- [ ] Конфліктні дні (той самий чистий `find_weather_conflicts`, ті самі пороги
      з `Settings`) підсвічені — узгоджено з тим, що запропонував би
      `weather_plan_job`.
- [ ] Best-effort: нема локації або Open-Meteo впав → сторінка рендериться без
      чіпів (патерн live-fallback `_strength_details`).
- [ ] Один fetch на рендер (`run_in_threadpool`), без кешу в v1 — Open-Meteo
      безключовий і дешевий; якщо латентність сторінки помітна, short-TTL кеш —
      окремим кроком.
- [ ] Тест роутера: з мокнутим форкастом чіпи в HTML; з `None` — сторінка 200 без
      чіпів.

## Підводні камені

- Не запускати з веб-сторінки Claude-виклик погодної адаптації — сторінка лише
  *показує*; пропозиції лишаються за джобою (одна пропозиція за раз,
  `_has_pending_proposal`).
- `find_weather_conflicts` чистий і мережево-вільний — переюзати як є, не
  дублювати порогові правила в шаблоні.
