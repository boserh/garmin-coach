# CODE-05 · Спільний report-флоу для бота, вебу і morning

**Тип:** рефакторинг · **Оцінка:** S–M · **Пріоритет:** середній · **Залежності:**
зручно зробити разом зі ST-03 (погода в on-demand) — вона чіпає ті самі три місця

## Проблема

Той самий конвеєр «payload → (погода) → run_analysis → stale-note → текст»
зібраний тричі з дрібними розбіжностями:

- `bot/handlers.py::report` (~:77–86) — `_REPORT_STALE`-примітка + reply;
- `app/routers/reports.py::report_json` (~:42) — своя `_STALE_NOTE` у JSON;
- `bot/jobs.py::_deliver_morning` (~:89–114) — своя `_MORNING_STALE` + префікси.

ST-03 (погода) вимагатиме правити всі три; EP-07/EP-08 додадуть четвертого
споживача. Розбіжності в текстах stale-приміток уже є і не задокументовані.

## Acceptance criteria

- [ ] Один хелпер (напр. `app/analysis/delivery.py::build_report(session, user, *,
      question=None, weather=None) -> ReportResult`) повертає `text`,
      `synced_today`, `last_data_date` — а канал сам вирішує, як показати
      stale-примітку (Telegram-текст vs JSON-поле).
- [ ] Три виклики переведені; тексти приміток зведені в одне місце
      (константи поряд із хелпером), відмінності morning/on-demand — свідомі
      й закоментовані.
- [ ] Морнінг-специфіка (once-a-day guard, вікно, MFA-notify, префікси
      /test_morning) лишається в `bot/jobs.py` — хелпер не знає про Telegram.
- [ ] Дедуп-кеш поведінка незмінна (ключі не чіпаються — хелпер лише збирає
      виклик, який і так існує).
- [ ] Тести роутера `/report.json` зелені без змін контракту відповіді.

## Підводні камені

- Не переускладнити: це extraction, не новий шар абстракції. Якщо результат
  вимагає більше трьох параметрів конфігурації — зупинитись і лишити два
  простіші хелпери.
