# EP-16 · Сезонна періодизація (кілька стартів, макроцикли)

**Тип:** епік · **Оцінка:** XL · **Пріоритет:** довгостроковий (коли з'явиться
другий старт у сезоні) · **Залежності:** EP-02 ✅ (адаптація), EP-05 (race pack —
природно вписується в taper-фазу)

## Сторя

> Як бігун із сезоном (весняна десятка → осінній півмарафон), я хочу задати
> **всі** старти з пріоритетами (A/B/C), щоб тренер збудував макроплан сезону —
> base → build → peak → taper → відновлення → наступний блок — а не окремі
> незв'язані плани, де кожен новий стартує «з нуля».

## Контекст

Зараз `TrainingPlan` — один активний план під одну ціль; після перегонів юзер
архівує і генерує наступний, без пам'яті про попередній блок (прогрес видно
тільки через fitness-снапшот). Періодизація — те, що відрізняє «план на 8 тижнів»
від «тренера на рік», і головна структурна перевага перед Runna-стилем.

## Скоуп

- Нова сутність `Season` (user, назва, список стартів: дата/дистанція/ціль/
  пріоритет A/B/C) над `TrainingPlan` — план стає **блоком** сезону
  (`season_id`, `phase`: base/build/peak/taper/recovery).
- **Генерація дворівнева**: (1) Opus будує макроскелет сезону — фази, тижневі
  діапазони обсягу, B/C-старти як «жорсткі тренування» всередині блоків;
  (2) детальні `PlannedWorkout` генеруються **тільки на поточний блок** тим самим
  наявним шляхом (`run_plan_generation` + контекст макроплану) — довгі детальні
  плани далеко наперед не мають сенсу через адаптацію.
- Перехід між блоками: наприкінці блоку — автопропозиція «згенерувати наступний»
  з фактичними результатами попереднього в контексті (compliance з EP-01,
  дайджест-дані EP-07).
- Після A-старту — recovery-блок автоматично (reverse taper), не «пустота».
- Веб: `/plan` показує стрічку сезону (фази + старти) над поточним блоком;
  intake-форма отримує режим «сезон» поряд зі звичайним «один старт».

## Acceptance criteria

- [ ] Звичайний одноцільовий флоу не змінюється (сезон — опційний рівень зверху).
- [ ] Rolling-window push у Garmin працює без змін (пушиться активний блок —
      для plan_sync нічого нового).
- [ ] B-старт усередині блоку відображається в плані як сесія типу race
      (не ламає matching і дайджест).
- [ ] Зміна дати/пріоритету старту → перегенерація тільки майбутніх блоків,
      історія недоторкана.
- [ ] Вартість під контролем: макроскелет — один Opus-виклик на сезон;
      блоки — як звичайна генерація.

## Підводні камені

- Найбільший скоуп-ризик у беклозі: різати строго — v1 це «список стартів +
  макроскелет + контекст для генерації блоку», без автопереходів і recovery-магії.
- Схема `GeneratedPlan` і промпти вже великі — макрорівень тримати **окремим**
  промптом/схемою (`SYSTEM_SEASON`), не роздувати `SYSTEM_PLAN`.
- UX-пастка: не змушувати юзера з одним стартом проходити «сезонний» майстер.
