# PERF-01 · Паралелізація per-user джоб бота

**Тип:** перфоманс · **Оцінка:** M · **Пріоритет:** високий при >5–10 юзерів ·
**Залежності:** немає (синергія з PERF-05: ліміт конкурентності до Garmin)

**❄️ Frozen (2026-07):** за 1–2 реальних юзерів болю немає; тригер розморозки — >5 юзерів (ANALYSIS.md §1.3).

## Проблема

`morning_job`, `plan_sync_job` і `plan_adapt_job` (`bot/jobs.py`) ітерують юзерів
**послідовно** в одному циклі і через **одну** спільну `async_session_maker()`-сесію.
Один повільний юзер блокує всіх наступних:

- `user_runtime` → garth login може висіти до ~25с (MFA-гейт у `app/garmin/mfa.py`);
- `build_payload_cached` для юзера з "дірками" в історії — десятки Garmin-запитів;
- Claude-виклик морнінг-звіту — ще кілька секунд на юзера.

При 20 юзерах ранковий тік легко виходить за 20-хвилинний інтервал повторення джоби —
тіки почнуть накладатись, а останні юзери отримають звіт ближче до полудня.

## Acceptance criteria

- [ ] Per-user обробка йде через `asyncio.gather` з `asyncio.Semaphore` (ліміт ~3–5).
- [ ] **Окрема DB-сесія на юзера** (зараз одна сесія шариться між усіма ітераціями —
      при паралелізації це перше, що зламається: AsyncSession не є concurrency-safe).
- [ ] Помилка одного юзера не валить і не блокує інших (зараз так — зберегти).
- [ ] Порядок логів залишається читабельним (лог-рядки вже містять `user={id}`).
- [ ] Тест: 3 фейкові юзери, один "повільний" (sleep) — інші два завершуються раніше.

## План по файлах

- `bot/jobs.py` — спільний хелпер `_run_per_user(recipients, worker, limit=N)`:
  на кожного юзера власний `async with async_session_maker()`, семафор, gather.
  Переиспользовати в `morning_job`, `plan_sync_job`, `plan_adapt_job`.
- `tests/` — тест хелпера з фейковим worker.

## Підводні камені

- SQLite: паралельні писателі → "database is locked" (див. PERF-03). Ліміт семафора
  на SQLite тримати =1..2, на Postgres піднімати.
- Ліміт конкурентності — це і захист Garmin від сплеску запитів (PERF-05):
  не піднімати вище 3–5 навіть на Postgres без rate limiter'а.
