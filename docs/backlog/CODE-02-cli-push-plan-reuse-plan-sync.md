# CODE-02 · CLI `push-plan` поверх `plan_sync` (прибрати дубль rolling window)

**Тип:** рефакторинг · **Оцінка:** S · **Пріоритет:** середній · **Залежності:** немає

## Проблема

`app/cli.py::_push_plan` (~:180–245) власноруч відбирає "upcoming planned у вікні
--days" і пушить через `push_workout` — та сама логіка вікна/pushable, що й у
`app/garmin/plan_sync.sync_plan_to_garmin` (forward-пас). Дві копії правил
"що вважати пушабельним" уже одного разу розійшлися б на силових
(`_pushable` у plan_sync знає про `strength_plan`/`garmin_template_id` — CLI-фільтр
треба щоразу згадувати оновити).

## Acceptance criteria

- [ ] Forward-відбір (вікно, статуси, pushable, skip уже запушених) живе в
      одному місці — `plan_sync` — і параметризується (`days`, `only_date`).
- [ ] CLI `push-plan` кличе спільний шлях; свої лишаються тільки CLI-специфіки:
      `--dry-run` (друк payload'ів без запису) та ігнорування
      `garmin_sync_enabled` (явний manual override — поведінку зберегти!).
- [ ] `unpush-plan` так само спирається на спільні хелпери remove-шляху.
- [ ] Ідемпотентність і толерантність до "workout уже видалений в UI" —
      без змін (є тести/ручний прогін).

## Підводні камені

- `--dry-run` вимагає віддати з спільного шляху "що б я запушив" без побічних
  ефектів — найпростіше: спільна функція відбору повертає список workout'ів,
  а пуш-цикл лишається тонким і в CLI, і в plan_sync.
- Не змінити семантику `--date` (один день) — вона використовується в
  документованих сценаріях CLAUDE.md.
