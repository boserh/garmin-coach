# EP-15 · Рельєф і grade-adjusted pace (GAP)

**Тип:** епік · **Оцінка:** M–L · **Пріоритет:** середній (must-have для горбистих
маршрутів/трейлу, інакше аналіз темпу бреше) · **Залежності:** немає

**Перетини (ANALYSIS.md §1.6):** GAP-функцію писати реюзабельним модулем — на неї чекає EP-05 фаза 2 (сегментний пейсинг по рельєфу траси).

## Сторя

> Як бігун по горбистих маршрутах, я хочу, щоб аналіз пробіжки враховував набір
> висоти: «темп 6:10, але з поправкою на підйом це ~5:30 — інтенсивність за планом» —
> замість «ти біг повільніше плану», коли я просто біг угору.

## Контекст

Зараз `series` = `[{d, p, hr}]` — висоти нема, і аналіз (`_segments` →
`SYSTEM_ACTIVITY`) трактує сповільнення на підйомі як слабкість. Garmin
`/details` уже віддає altitude-колонку в тих самих metrics (той самий фетч,
descriptor-key lookup — механіка резолвінгу колонок готова в
`fetch_activity_series`); загальний elevation gain є і в activity summary.
FIT-файли експорту теж мають altitude → бекфіл старих пробіжок можливий
(`import-fit-series` уже парсить records).

## Скоуп

- `series` → `[{d, p, hr, e}]` (висота, м): live-шлях (`fetch_activity_series`)
  + бекфіл через розширення `import-fit-series` (тільки додати поле, матчинг
  готовий).
- GAP: стандартна поправка темпу на градієнт (формула Minetti/Strava-подібна,
  проста функція від grade) — рахується по сегментах у `_segments`;
  у payload аналізу йдуть і сирий, і скоригований темп + набір/спад на сегмент.
- `SYSTEM_ACTIVITY` — інструкція трактувати виконання відносно GAP, коли набір
  значний (>10 м/км), і не згадувати рельєф на пласких пробіжках.
- Матчинг план/факт (EP-01): «виконано за темпом» на горбистому маршруті
  оцінюється по GAP, не по сирому темпу.
- Веб-деталка: третій спарклайн (профіль висоти) під pace/HR — той самий
  SVG-механізм `_run_charts`.

## Підводні камені

- Барометрична висота шумить — перед розрахунком градієнта згладити (rolling
  mean по 3–5 точках), інакше GAP стрибає на кожній точці.
- Downsample до ~150 точок згладжує короткі круті гірки — сумарний gain брати
  з activity summary (точний), GAP по сегментах — як наближення; не обіцяти
  точність трейл-ультра.
- Кеш `series:v1:<id>` — нове поле означає bump до `series:v2` (інакше старі
  кешовані серії без висоти вважатимуться повними).
