# ST-08 · Валідувати назви вправ (не лише категорії) перед пушем на годинник

**Тип:** покращення коректності · **Оцінка:** S · **Пріоритет:** середній (реальна
дірка у ланцюжку «LLM → годинник») · **Залежності:** —

## Проблема

`repository._sanitize_strength` і swap-гілка `apply_plan_ops` валідують **тільки**
`category` через `exercises.valid_category` — поле `exercise` (конкретна вправа
всередині категорії) зберігається як `ex.upper()` без жодної перевірки
(`app/garmin/repository.py:641`). Галюцинована назва вправи з валідною категорією
проходить до `strength_plan`/`exercise_edits` і далі на годинник — Garmin покаже
порожній/зламаний крок.

При цьому інструмент уже є і **ніде не викликається**:
`exercises.valid_exercise(category, exercise)` (перевіряє назву проти варіантів
категорії з `exercise_catalog.json`), як і `exercises_for(category)` — обидві
функції зараз мертвий код (`app/garmin/exercises.py:76`, `:87`).

## Acceptance criteria

- [ ] `_sanitize_strength`: якщо `exercise` задано і не проходить
      `valid_exercise(cat, ex)` — **скинути назву в `None`** (крок лишається
      категорійним — це валідно для Garmin), не викидати вправу цілком.
      Залогувати один раз: `EXERCISE invalid: <CAT>/<NAME>` (за зразком
      `EXERCISE unmapped` в `exercise_names.py`).
- [ ] Та сама перевірка для `op.exercise` у swap-гілці `apply_plan_ops`
      (`swap_exercise` → `exercise_edits`).
- [ ] Тести: валідна пара проходить; невалідна назва при валідній категорії →
      `exercise=None`, категорія збережена; невалідна категорія → вправа
      викинута (поточна поведінка не змінюється).
- [ ] `exercises_for` або отримує використання (наприклад, підказка варіантів
      у `SYSTEM_PLAN_EDIT`-контексті), або видаляється — не тримати мертвим.

## Підводні камені

- `exercise=None` — легітимний стан (категорійний крок): валідація не повинна
  вимагати назву, лише відхиляти неправильну.
- Якщо `exercise_catalog.json` відсутній, працює `_FALLBACK_CATEGORIES` **без
  варіантів** — у цьому режимі `valid_exercise` не має бази для перевірки назв:
  пропускати валідацію назви (не скидати все в `None`), інакше деградація без
  каталогу стане руйнівною.
- Не чіпати кеш/пуш-шлях: зміна тільки в точці запису (`repository`), як і
  поточна валідація категорій.
