# PERF-05 · Per-user fetch-lock і rate limit до Garmin

**Тип:** перфоманс/стійкість · **Оцінка:** M · **Пріоритет:** високий разом з PERF-01
(без нього паралелізація множить запити до Garmin) · **Залежності:** немає

## Проблема

1. **Подвійний фетч того самого юзера**: `build_payload_cached` не має per-user
   guard'а — ранковий тік і одночасний `/report` (або два веб-запити) для одного
   юзера паралельно тягнуть ті самі дні з Garmin і навперебій upsert'ять їх у БД.
   Марні запити + зайві шанси на 429.

2. **Немає глобального ліміту до connectapi**: N юзерів × (login + дні + activities
   + planned + series) без жодного тротлінгу. Garmin уже відомий 429-ми (саме тому
   існує offline `import-export`). Один 429 зараз просто падає в лог конкретного
   юзера; шторм 429 вранці — усі без звітів.

## Acceptance criteria

- [ ] Per-user `asyncio.Lock` навколо fetch-фази `build_payload_cached`
      (`WeakValueDictionary` user_id → Lock): другий конкурентний виклик чекає
      і потім читає вже свіжу БД (сьогоднішній день можна перечитати дешево).
- [ ] Глобальний rate limiter на `client._api`-рівні (token bucket, ~2–4 rps на
      процес, конфіг через Settings) — один на всі Garmin-виклики процесу.
- [ ] Обробка 429: розпізнаваний виняток + експоненційний backoff з 2–3 ретраями
      всередині limiter'а; після вичерпання — поточна поведінка (лог + skip юзера).
- [ ] Морнінг-тік із 5 юзерами не перевищує ліміт (перевірити лічильником у тесті
      з фейковим провайдером).

## План по файлах

- `app/garmin/service.py` — per-user lock у `build_payload_cached`.
- `app/garmin/client.py` — rate limiter + backoff у низькорівневому фетчі
  (місце одне, бо всі виклики йдуть через нього).
- `app/core/config.py` — `GARMIN_RPS`, `GARMIN_RETRIES` з дефолтами.
- `tests/` — конкурентні виклики payload для одного юзера → один прохід фетчу.

## Підводні камені

- Lock має жити навколо **фетчу**, не навколо всього payload-шляху — інакше
  повільний Garmin блокує і суто-БД читання.
- Limiter — синхронний код (клієнт працює в threadpool): `threading.Semaphore` +
  sleep, не asyncio-примітиви.
- Не чіпати MFA-шлях: 25-секундне очікування гейта — не "повільний запит",
  його не можна ретраїти.
