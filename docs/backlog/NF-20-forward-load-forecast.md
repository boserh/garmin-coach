# NF-20 · Прогноз навантаження вперед: «що зробить із ACWR цей тиждень плану»

**Тип:** нова фіча (BA-аналіз 2026-07-23, кат. C) · **Оцінка:** M ·
**Пріоритет:** середній · **Поверхня:** обидва (блок на /plan і дашборді; контекст
адаптації) · **Залежності:** NF-05 ✅ (шкала навантаження), NF-11 ✅ (оцінка тривалості)

## Проблема

ACWR/acute load від Garmin — суто ретроспектива: показує, що ти **вже** перевантажився.
План на 7 днів уперед ніхто не перевіряє проти навантаження числом — SYSTEM_PLAN_ADAPT
«інтуїтивить» з ретро-даних. Користувач, який вручну доредагував план чатом (+2 сесії),
не бачить, що субота виведе його в червону зону, поки Garmin не покаже це постфактум —
а NF-04/NF-09 вже потім розгрібають.

## User story

Як користувач я хочу бачити на сторінці плану прогноз «якщо виконаю все заплановане,
навантаження тижня буде X (+22% до звичного, ACWR ≈1.4)», щоб ловити перевантаження
до того, як воно станеться, а не після.

## Обсяг

**Входить:** чистий модуль `app/loadforecast.py` (нуль LLM): TRIMP-оцінка планової
сесії = оцінена тривалість (`fueling.estimate_minutes` — уже вміє steps/dist/тип) ×
ваговий коефіцієнт типу (mapping type→зона в дусі `multisport`-шкали: easy/recovery
1.5, long 2, tempo 3, intervals 4, strength 2, cycling по hr_zone або 2); прогнозний
тиждень = сума по майбутніх `planned`-сесіях + факт минулих днів тижня
(`weekly_activity_load`); ACWR-прогноз = прогнозний acute (7 дн уперед) / хронічний
(28 дн назад, факт). Подача: блок на `/plan` («Прогноз тижня: 412 (+22%), ACWR
≈1.4 ⚠️» з підписом «оцінка»), рядок на дашборді, число в контекст
`run_plan_adaptation` (не кешується — без кеш-ключів).

**Не входить:** блокування редагувань плану за прогнозом (тільки показ); окрема
команда бота; калібрування коефіцієнтів по юзеру (v1 — фіксована шкала, чесно
підписана як оцінка).

## Acceptance criteria

- [ ] Прогноз рахується з планових сесій + факту поточного тижня; скасована
      (skipped) сесія одразу знижує прогноз.
- [ ] ACWR-прогноз ≥`FORECAST_ACWR_WARN`=1.4 → жовта позначка, ≥1.6 → червона;
      немає 28 днів історії → «калібруюсь», нуль чисел.
- [ ] Контекст адаптації містить `load_forecast` (тест: LLM мокнутий, число в
      контексті).
- [ ] Чат-едіт плану (add/skip) міняє прогноз при наступному рендері /plan.
- [ ] Тести модуля: ваги, змішаний тиждень факт+план, порожній план.

## Ризики

- Груба оцінка TRIMP планової сесії (без фактичного HR) — головна причина
  Confidence 0.6; підпис «оцінка» всюди, пороги консервативні, коефіцієнти в
  одному місці для майбутнього калібрування (кандидат: підганяти по власних
  фактичних TRIMP vs планова оцінка).
