# OPS-08 · Моніторинг свіжості бекапів БД

**Тип:** надійність (BA-аналіз 2026-07-23, кат. B) · **Оцінка:** S ·
**Пріоритет:** середній (низька частота, критичний наслідок) · **Поверхня:** обидва
(веб /status для адміна; бот — DM адміну) · **Залежності:** OPS-02 ✅ (сам бекап)

## Проблема

OPS-02 закрив бекапи (нічний systemd-таймер + off-SD rsync), але ніхто не стежить,
що вони **продовжують** відбуватись: згаслий таймер, відвалений USB, повний диск —
і про мертві бекапи дізнаєшся в момент відновлення, тобто найгіршого дня. Класична
друга половина бекап-історії: «бекап без моніторингу — це лотерея». Вік останнього
файлу зараз видно лише по ssh + `ls backups/`.

## User story

Як адмін я хочу бачити вік останнього успішного бекапу і отримати повідомлення, якщо
бекапи не робляться кілька діб, щоб SD-смерть Pi не стала втратою річної історії.

## Обсяг

**Входить:** `scripts/backup_db.py` після успіху пише маркер `backups/last_ok.json`
(ts, шлях, розмір, rsync ok/фейл); `/status` (для `is_admin`) віддає
`backup_age_hours` + `backup_rsync_ok`; morning-тік адміну: DM «⚠️ Бекапу немає
N діб» при віці > `BACKUP_WARN_DAYS`=3 (guard раз/добу через `bot_state`). Рядок
на дашборді адміна.

**Не входить:** перевірка відновлюваності (restore-тест — ручна процедура OPS-02);
моніторинг вмісту бекапу; Postgres-гілка (frozen разом з PERF-03).

## Acceptance criteria

- [ ] Успішний бекап оновлює маркер; фейл rsync фіксується в маркері окремо від
      фейлу самого бекапу.
- [ ] `/status` адміна показує вік; не-адмін цього поля не бачить.
- [ ] DM іде раз/добу лише адмінам, лише при перевищенні порога; свіжий бекап
      зупиняє нагадування без ручних дій.
- [ ] Відсутність каталогу/маркера (бекапи ще не налаштовані) → чесний
      `backup_age_hours: null`, DM не спамить щодня, а раз на `BACKUP_WARN_DAYS`.
- [ ] Тести: парс маркера, пороги, guard.

## Технічні дотики

- `scripts/backup_db.py` (маркер), `app/routers/health.py::status` (admin-гілка),
  `bot/jobs.py` (перевірка в тіку — чистий stat файлу, нуль мережі),
  `app/core/config.py::BACKUP_WARN_DAYS`.

## Ризики

- Маркер на тій самій SD, що й БД — так, але сигнал «маркер старий» працює і коли
