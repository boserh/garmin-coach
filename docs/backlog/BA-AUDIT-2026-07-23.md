# BA-аудит 2026-07-23 — usability, спостережуваність, нові фічі

Продуктовий аналіз застосунку (джерело правди — **код**, не README; розбіжності —
у §6). Результат — 19 тікетів трьох категорій: **A** usability наявного
(ST-15…ST-21, окремий фокус — керування даними активностей), **B** спостережуваність
і надійність (OPS-04…OPS-09), **C** нові продуктові фічі (NF-17…NF-22).
Обмеження враховані в кожному тікеті: фіксований стек, RPi 4, вартість LLM,
неофіційний Garmin API, спільне ядро без дублювання бот/веб, серверний веб-UI.

## 2. Сценарний аналіз — де болить

**Онбординг/Garmin-підключення.** Стан авторизації видно лише як live-логін-проба в
`/status`; які ендпоінти падали, чи деградує API — невидимо (лог на Pi). → OPS-05.
Ручного «синкни зараз» немає ніде, крім прихованого `/test_morning`. → ST-15.

**Ранковий звіт.** ⚠️ [Скориговано §7] Первісно тут стверджувалось «будь-який збій
вікна → тиша» — насправді наявний stale-звіт о deadline покриває головний кейс
(«годинник не синкнувся»), Claude-помилка шле текст, MFA має свій DM. Реально «тиха»
лише вузька щілина — жорсткий Garmin-збій логіну о deadline. → OPS-05 (осн.), OPS-07
(вузький залишок), OPS-04 (детальна причина). Гарди (`bot_state`) скидаються лише
руками адміна через `/ui/bot_state/delete`.

**Перегляд активності.** Немає: ресинку однієї активності (`upsert_activity`
оновлює лише вікно останніх N), пробиття битого 365-денного кешу series/sets,
приховання дубля/битого треку (отруює рекорди EP-14, бюджет NF-05, матчинг EP-01),
перегенерації аналізу (`ActivityRecord.analysis` + 7-денний dedup), ознаки «дані
неповні». → ST-15, ST-16, ST-17, ST-19, ST-18.

**Щоденні метрики.** `persist_payload` зберігає день при `has_data=True`, навіть
якщо синк частковий; `build_payload_cached` ніколи не перефетчує збережені минулі
дні — частковий день застигає назавжди, ремонт лише офлайн-`import-export`. Юзер
дір не бачить. → ST-18 (+ST-15 як ручний escape hatch).

**План і календар Garmin.** Помилка пушу тиха, `/plan` не показує «чи на
годиннику», ручного синку з UI немає (лише CLI). → OPS-09. Кривий матч план/факт
або сесія без годинника не виправляються з UI (`status`/`completed_activity_id`
без ручки) — спотворюють compliance → дайджест і адаптацію. → ST-21.

**Бекфіл із GDPR-експорту.** CLI-only за задумом (ок для разової операції), але
після нього немає видимого «скільки днів усе ще неповні» — знову ST-18.

**Витрати.** `/costs`, `month_cost` на дашборді, `cached`-share — є. Небачений
хвіст: hit-rate по kind і розмір кешів (→ OPS-06), вартість force-перегенерацій
позначена в UI (→ ST-19 AC).

**Фонові джоби взагалі.** Шість джоб — нуль видимої історії запусків. → OPS-04.
Бекапи йдуть, але свіжість ніхто не стереже. → OPS-08.

**Бот↔веб.** Основні петлі вже спільні (EP-11: chat/confirm через `bot_state`).
Нових розривів не плодимо: кожен тікет називає поверхню; дані-контроль — веб
(форми) + тонкі бот-команди.

## 4. Топ-5 до негайної реалізації

1. **NF-17** — найдешевше замикання вже збудованої петлі: NF-10 має verdict-механіку,
   бракує одного поля форми. Максимум цінності (число до цілі в /goal, дайджесті й
   генерації плану) за ~2.5 дня, нуль нових LLM-викликів.
2. **OPS-05** — неофіційний API — екзистенційний ризик №1 проєкту (OPS-01), а його
   деградація сьогодні видима лише через grep на Pi. Дає ранній сигнал міграції
   (і поглинає вузький залишок OPS-07).
3. **ST-18** — найчастіший «тихий» дефект даних: частково синкнутий день застигає
   назавжди і псує базлайни/алерти/тренди, на яких стоїть половина фіч. Плюс перший
   крок battery «керування даними» (спільні механіки зі ST-15/16).
4. **NF-19** — найбільша нова цінність з наявних даних: series роками лежать у БД
   без вжитку між активностями; «швидше за той самий пульс» — чесніший прогрес,
   ніж Garmin-прогнози, нуль LLM-витрат.
5. **OPS-04** — фундамент спостережуваності: 6 фонових джоб не лишають видимого сліду.
   Розблоковує детальні причини для OPS-05/07/08/09 і закриває питання «чи запускалось
   і чим скінчилось» без ssh.

(ST-15 — одразу наступний у черзі: розблоковує ST-16/ST-19 і є escape hatch для
ST-18-старих днів; у топ-5 не вліз лише через строгий порядок за score. NF-20 — на
одному score з OPS-04, але нижча Confidence через грубу оцінку планового TRIMP.)

## 5. Не робити зараз

- **SSE-стрімінг чату (EP-11 AC)** — вимагає переходу на AsyncAnthropic проти
  свідомого PERF-04b-рішення; цінність на 1–3 юзерів мала. Лишається відкладеним.
- **Сезонні/множинні вікна базлайнів (NF-01 ext)** — поки нема двох повних сезонів
  чистої історії, сезонні персентилі — шум; 90-денне вікно достатнє.
- **Closed-loop прогресія силових (EP-03 ext)** — заблоковано ззовні: Garmin не
  віддає виконані сети/повтори; чекати на endpoint, не будувати обхідні милиці.
- **EP-06 квоти / Postgres (PERF-03) / відкриття `/register`** — заморожені
  свідомо: чужі юзери на API, який Garmin активно закриває, — не зараз.
- **Автовиявлення дублів активностей** — рідкісний кейс; ручний ST-17 покриває
  95% болю за 10% складності.
- **Prometheus/Grafana-метрики** — оверкіл для Pi з 1–3 юзерами; OPS-04/05/06
  дають потрібну видимість у наявному веб-UI.
- **Web-push/PWA-нотифікації** — Telegram уже є каналом пушів; другий канал =
  подвійна логіка guard-ів без нової цінності.
- **Пост-рейс аналіз забігу** — цінний, але 1–2 рази/рік і потребує окремого
  дизайну (спліт-аналіз vs план з race pack); кандидат наступної партії після
  NF-22.
- **OPS-07 як окрема фіча** (додано після ревізії §7) — наявний stale-звіт о deadline
  уже покриває головний кейс; реальний залишок вузький і поглинається OPS-05. Не
  робити окремо — або в межах OPS-05, або зовсім опустити.
- **NF-18, якщо на власній історії немає патерну «пропуски-через-хворобу»** (§7) —
  диференціація від NF-09 реальна, але спрацьовує рідко; спершу подивитись, чи такий
  епізод узагалі трапляється, і лише тоді будувати.

## 7. Пост-PR ревізія (звірка тікетів із кодом після мержу #173)

Прискіплива повторна звірка створених тікетів проти коду виявила три дефекти —
виправлені цим правочним PR (гілка перезапущена від `main`, бо #173 уже змерджано):

1. **OPS-07 — «Проблема» була значною мірою хибною.** Первісне формулювання
   («будь-який збій ранкового вікна → тиша») ігнорувало наявний deadline-механізм:
   `_deliver_morning` (`bot/jobs.py:181,188,194`) о 12:00 усе одно шле звіт зі
   stale-приміткою `_MORNING_STALE`, тобто нотифікація «годинник не синкнувся» вже
   існує. Плюс Claude-помилки шлють текст помилки, MFA має свій DM, Telegram
   ретраїться. Реальний залишок — лише жорсткий Garmin-збій логіну о deadline (до
   побудови payload), який до того ж покриває OPS-05. Тікет пере-скоуплено на цю
   вузьку щілину, RICE 3.6 → 0.7, рекомендація «злити в OPS-05 або опустити».
2. **OPS-09 — фактична помилка.** Тікет називав `sync_plan_to_garmin` «мовчазною» —
   насправді вона **вже повертає** `{"pushed": n, "removed": n}`
   (`app/garmin/plan_sync.py:140,167`). Прогалина вужча: dict відкидається
   викликачами й не має деталей помилок. «Проблему»/«Технічні дотики»/«Обсяг»
   виправлено на «розширити наявний підсумок + почати його зберігати», RICE без змін.
3. **NF-18 — слабка диференціація від NF-09.** Первісний тригер (стек health-алертів)
   майже повністю дублював наявний `_deload_check_for_user` (`bot/jobs.py:416`).
