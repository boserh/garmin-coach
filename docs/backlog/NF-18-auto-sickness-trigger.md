# NF-18 · Автотригер «схоже, захворів» → пропозиція перебудови блоку (/sick без /sick)

**Тип:** нова фіча (BA-аналіз 2026-07-23, кат. C) · **Оцінка:** M ·
**Пріоритет:** низький (рідко, але дорого, коли трапляється) · **Поверхня:** бот ·
**Залежності:** NF-03 ✅ (движок), EP-08 ✅ (детектор), NF-09 ✅ (патерн deload-тригера)

## Проблема

NF-03 сам називає це майбутнім тригером: «автоматичного EP-08-style детектора хвороби
ще немає — `/sick` єдина точка входу». Реальний хворий юзер `/sick` не пише: він
лежить з температурою, план сиплеться в missed, а недільна адаптація потім «лікує»
наслідки точковими move/modify не з тієї причини. Всі складові вже є: EP-08 детектить
стійкі аномалії відновлення, NF-09 показав патерн «детектор → готова пропозиція
з ✅/❌», NF-03 дав безпечний sick-движок з `_filter_sick_ops`.

## User story

Як користувач я хочу, щоб коли моє відновлення кілька днів поспіль виглядає як
хвороба, бот сам запитав «перебудувати блок у щадному режимі?», щоб план полагодився
одним тапом навіть коли мені не до команд.

## Обсяг

**Входить:** правило в morning-тіку (після `_deload_check_for_user`, з тим самим
пріоритетом «одна risk-пропозиція на день»): `build_health_alerts` показує ≥2
одночасні алерти з набору {hrv_low, rhr_up, sleep_debt} упродовж
`SICKNESS_STREAK_DAYS`=2 днів поспіль → одна DM «Схоже на хворобу або сильну
перевтому. Перебудувати найближчий блок у щадному режимі?» з ✅/❌; ✅ →
`run_sick_check` (той самий, що `/sick`) → стандартні кнопки підтвердження опсів;
❌ → snooze `SICKNESS_GUARD_DAYS`=7 (guard у `bot_state`). Гейти: `alerts_enabled`
+ `plan_adapt_enabled` + активний план + `_has_pending_proposal`.

**Не входить:** автозастосування без підтвердження (ніколи); діагностика причин
(формулювання підкреслено немедичні — успадковано від NF-03); тригер із
travel-сценарію (нема сигналу подорожі в даних).

## Acceptance criteria

- [ ] Дводенний стек ≥2 алертів → одна DM-пропозиція; одиночний алерт або 1 день →
      тиша (тест на межі).
- [ ] ✅ жене `run_sick_check` і показує звичні кнопки опсів; ❌/ігнор → мовчання
      7 днів; повторний стек після guard → нова пропозиція.
- [ ] Жодного Claude-виклику до явного ✅ (тест: детектор чистий).
- [ ] Пропозиція не виникає, коли вже висить інша (`_has_pending_proposal`).
- [ ] Деактивований/вимкнений юзер → тиша.

## Технічні дотики

- `bot/jobs.py`: `_sickness_check_for_user` (после deload-check) + callback
  `sickness:yes/no` у `bot/handlers.py` (реюз `pending_plan`-флоу як `/sick`);
- `app/health.py` (нічого — читаємо готовий `HealthReport`), `app/core/config.py`
  (`SICKNESS_STREAK_DAYS`, `SICKNESS_GUARD_DAYS`).

## Вартість LLM / запити до Garmin

LLM: 0 у детекторі; 1 Sonnet plan-ops виклик **лише після явного ✅** (~$0.02–0.05;
plan-ops свідомо не кешуються — контекст щоразу інший, кеш тут не поможе й не
потрібен: епізод — кілька разів на рік). Garmin: 0 (усе з БД).

## Ризики

- False positive «важкий тиждень ≠ хвороба» — головний; тому (а) питання, не дія,
  (б) консервативні пороги, (в) семиденний snooze після ❌. Валідувати пороги на
  власній історії перед вмиканням за замовчуванням.
