# NF-18 · Автотригер «схоже, захворів» → перебудова ПРОПУЩЕНОГО блоку (/sick без /sick)

**Тип:** нова фіча (BA-аналіз 2026-07-23, кат. C; тригер пере-визначено після звірки
з NF-09) · **Оцінка:** M · **Пріоритет:** низький (рідко, але дорого) · **Поверхня:**
бот · **Залежності:** NF-03 ✅ (sick-движок), EP-08 ✅ (детектор аномалій), NF-09 ✅
(deload-тригер — цей тікет свідомо його НЕ дублює)

## Проблема

NF-03 сам називає це майбутнім тригером: «автоматичного детектора хвороби ще немає —
`/sick` єдина точка входу». Реальний хворий юзер `/sick` не пише: він лежить із
температурою, план сиплеться в `missed`, а недільна адаптація потім «лікує» наслідки
точковими move/modify не з тієї причини.

⚠️ **Ключова відмінність від NF-09 (після звірки з кодом)**: наявний
`_deload_check_for_user` (`bot/jobs.py:416`) уже перетворює actionable-сигнал
injury/health на ✅/❌-пропозицію. Тож детектувати «стек health-алертів» тут — означало б
**дублювати NF-09**. Різниця, яку NF-09 **не бачить**, — це напрям дії:

| | NF-09 (deload) | NF-18 (sick) |
|---|---|---|
| Тригер | actionable health/injury + **важка сесія попереду** (`ADAPT_HEAVY_TYPES` у 5 дн) | **кілька `missed`-сесій поспіль** + аномалії відновлення |
| Що робить | полегшує **майбутнє** (deload вперед) | ремонтує **минуле** (skip пропущених, щадне повернення) |
| Вікно | `today..today+window` | `today-14..today+14` (`SICK_LOOKBACK`/`SICK_WINDOW`) |
| `adjust_level` | шанує (bounds діють) | свідомо ігнорує (хвороба — поза нормальними межами) |

Тобто NF-18 має спрацьовувати саме тоді, коли NF-09 **не** спрацює: важкої сесії
попереду може й не бути (юзер уже пропустив пів тижня), а блок усе одно треба
перебудувати назад, а не полегшити вперед. Сигнал «кілька `missed` поспіль» —
об'єктивний і NF-09-у недоступний (він дивиться лише вперед на важкі сесії).

## User story

Як користувач я хочу, щоб коли я пропустив кілька запланованих сесій поспіль **і**
моє відновлення виглядає як хвороба, бот сам запитав «перебудувати блок у щадному
режимі?», щоб план полагодився одним тапом навіть коли мені не до команд.

## Обсяг

**Входить:** правило в morning-тіку **після** `_deload_check_for_user` (і тільки якщо
той НЕ вистрілив — спільний «одна risk-пропозиція на день» гейт, `INJURY_WARNED_KEY`):
тригер = **≥`SICKNESS_MISSED_DAYS`=3 послідовні `missed`-сесії плану за останні 7 днів**
(об'єктивний сигнал, недоступний NF-09) **І** `build_health_alerts` actionable (≥1
recovery-аномалія — м'якший поріг, бо пропуски вже несуть основне навантаження
доказу). → одна DM «Схоже, ти вибув із графіка (N пропущених сесій) на тлі просілого
відновлення. Перебудувати найближчий блок у щадному режимі?» з ✅/❌; ✅ →
`run_sick_check` (той самий, що `/sick`, з `missed_days`≈довжина серії) → стандартні
кнопки підтвердження опсів; ❌ → snooze `SICKNESS_GUARD_DAYS`=7. Гейти: `alerts_enabled`
+ `plan_adapt_enabled` + активний план + `_has_pending_proposal` + `not deload_sent`.

**Не входить:** тригер лише від health-стеку без пропусків (це територія NF-09 —
не дублювати); travel-сценарій (нема сигналу подорожі в даних); автозастосування
без ✅; діагностика причин (формулювання підкреслено немедичні, успадковано з NF-03).

## Acceptance criteria

- [ ] Тригер вимагає **обидва**: серію ≥3 `missed` поспіль І actionable health-report;
      відсутність будь-якого → тиша (тест на межах).
- [ ] Коли того ж ранку вже вистрілив `_deload_check_for_user` — NF-18 мовчить (спільний
      guard), і навпаки; ніколи два risk-DM за день.
- [ ] ✅ жене `run_sick_check` з коректним `missed_days`; ❌/ігнор → мовчання 7 днів;
      новий стек пропусків після guard → нова пропозиція.
- [ ] Жодного Claude-виклику до явного ✅ (детектор — чистий DB-рід).
- [ ] Пропозиція не виникає при вже висячій іншій (`_has_pending_proposal`);
      деактивований/вимкнений юзер → тиша.

## Технічні дотики

- `bot/jobs.py`: `_sickness_check_for_user` (після deload-check, під тим самим
  `not deload_sent`-гейтом) + читання серії `missed` через
  `repository.recent_plan_workouts`/аналог (статуси за 7 днів); callback
  `sickness:yes/no` у `bot/handlers.py` (реюз `pending_plan`-флоу як `/sick`).
- `app/core/config.py` (`SICKNESS_MISSED_DAYS`, `SICKNESS_GUARD_DAYS`).
- `app/analysis/plans.py::run_sick_check` — уже приймає «скільки днів пропущено»
  (аргумент `/sick`), передати довжину серії.

## Вартість LLM / запити до Garmin

## Ризики

- False positive «пропустив з інших причин (відрядження, лінь) ≠ хвороба» — тому
  **обов'язкова** друга умова (health-аномалії), питання а не дія, 7-денний snooze.
- Перетин із NF-09 у кейсі «пропуски + важка сесія попереду»: спільний денний guard
  гарантує один DM; порядок (deload перший) означає, що forward-полегшення виграє,
  а sick-ремонт спрацьовує лише коли попереду важкого нема — саме потрібна семантика.
- Валідувати пороги на власній історії перед вмиканням за замовчуванням.
