# ST-21 · Ручне керування статусом сесії плану та матчингом план/факт

**Тип:** покращення (BA-аналіз 2026-07-23, кат. A) · **Оцінка:** M ·
**Пріоритет:** високий · **Поверхня:** веб (основна; бот уже має чат-опси) ·
**Залежності:** —

## Проблема

Матчинг EP-01 — best-effort: буває, легкий біг закриває темповик того ж дня, а сесія,
виконана без годинника (бігова доріжка без синку, чужий трекер), назавжди висне
`missed`. Виправити нема як: `PlannedWorkout.status`/`completed_activity_id` не має
жодного UI — лише чат-едіт (`/plan <текст>` вміє skip, але не «познач виконаним» і не
«відвʼяжи активність») або SQL. А статус живить усе: `weekly_compliance` → дайджест
«відстаєш», `_recent_compliance` → адаптація плану, `stepmatch` → калібрування темпів.
Один кривий матч тижнями зсуває пропозиції адаптації.

## User story

Як користувач я хочу на сторінці плану вручну виправити статус минулої сесії
(виконано/пропущено) і відвʼязати чи привʼязати активність, щоб compliance й
адаптація рахувались від правди, а не від помилки матчера.

## Обсяг

**Входить:** на `/plan` для минулих сесій — меню дій: «✅ виконано вручну»,
«⏭ пропущено», «↺ відвʼязати активність», «🔗 привʼязати…» (список активностей
±1 день від дати сесії); `match_info` помічається `{"manual": true}`; ручний статус
недоторканий для наступних прогонів `matching.match_activities`.

**Не входить:** редагування самих сесій (є чат-опси); ручний матч для strength
(v1 — лише біг/вело); зміни в боті.

## Acceptance criteria

- [ ] Ручна зміна статусу відображається одразу на `/plan`, у compliance
      (`weekly_compliance`) і в контексті наступної адаптації.
- [ ] «Відвʼязати» повертає сесію в `missed`/`planned` (за датою), чистить
      `completed_activity_id`/`match_info`; активність стає доступною для матчу
      іншій сесії.
- [ ] Автоматичний матчер не перезаписує рядок із `match_info.manual=true` (тест).
- [ ] «Привʼязати» пропонує лише власні активності сумісного типу в ±1 день.
- [ ] Тести: ізоляція по юзеру, ідемпотентність повторного тапу.

## Технічні дотики

- `app/routers/plan.py`: `POST /plan/workout/{id}/status` (+ рендер меню в
  `plan.html`).
- `app/garmin/repository/plans.py`: `set_workout_status(...)`, кандидати для
  привʼязки.
- `app/garmin/matching.py`: скіп рядків з `manual` (симетрично до наявного скіпу
  вже зматчених).

## Вартість LLM / запити до Garmin

0 / 0 — чисті DB-операції. Побічно точніший контекст адаптації = менше хибних
пропозицій (кожна — платний plan-ops виклик).

## Ризики

- Ручний статус vs пізній синк活 активності: якщо після «виконано вручну» приїде
  реальна активність — лишити ручний матч (manual переважає), задокументувати.

## RICE

Reach 4/міс · Impact 2 · Confidence 0.8 · Effort ~3 дні → **Score ≈ 2.1**
