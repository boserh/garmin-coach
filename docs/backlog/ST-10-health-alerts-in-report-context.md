# ST-10 · Health-алерти в контекст ранкового звіту

**Тип:** покращення (BA-аналіз 2026-07-21; задокументований future extension EP-08) ·
**Оцінка:** S · **Пріоритет:** високий (реальна неузгодженість двох каналів) ·
**Залежності:** EP-08 ✅, NF-01 ✅

## Проблема

Звіт і health-алерт живуть окремо: ранковий звіт може радити темповик у день, коли
активний `rhr_up`/`hrv_low`. NF-01-`norm` дає звіту сирі персентилі, але не
*висновок детектора* («метрика поза коридором уже 3+ дні»). EP-08 сам фіксує цю
синергію як «documented future extension».

## Acceptance criteria

- [ ] `run_analysis` (звіт/ранок, не `/deep`) кличе наявний
      `service.build_health_alerts` (той самий 90-денний зріз, нуль LLM) і при
      `level="alert"` кладе компактний `health_alerts` (kind/severity/detail) у
      `user_content` — у тому самому `user_id`-гейтованому блоці, що
      `norm`/`records`/`subjective`.
- [ ] `health_alerts` входить у `_cache_key` (наскрізна пастка README).
- [ ] Секція в `SYSTEM`: активний алерт → знизити тон рекомендацій, узгодити з
      порадою алерту; `calibrating`/`none` → ключа немає, промпт мовчить.
- [ ] Тести: контекст доїжджає в `analyze_with_stats`; ключ кешу міняється з
      появою алерту (`test_analysis.py`-патерн).

## Підводні камені

- Не дублювати обчислення: `build_health_alerts` уже читає ту саму історію, що
  `compute_baselines` для `norm`, — переюзати один рід історії на обидва,
  а не додати другий запит.
- Звіт не має перетворитись на другий алерт: алерт-DM (з cooldown'ом) лишається
  єдиним «попередженням»; звіт лише *узгоджується* з ним.
