# EP-14 · Особисті рекорди й віхи

**Тип:** епік · **Оцінка:** M · **Пріоритет:** середній (чистий retention/fun,
майже без LLM-витрат) · **Залежності:** немає; виграє від глибокої історії
(`import-export` уже вміє затягнути роки)

## Сторя

> Як бігун, я хочу, щоб бот помічав і святкував мої рекорди — найшвидші 5К/10К,
> найдовший біг, найбільший тиждень, новий максимум VO2max, покращений race
> prediction — і мав команду `/records` з усіма моїми бестами.

## Контекст

Уся сировина в БД: `activities` (дистанція/час/темп за роки), `weekly_run_volume`,
`extra` (VO2max, race predictions — щоденні значення). Garmin сам показує PR-и,
але мовчки і без контексту; ми можемо і привітати, і вписати рекорд у розповідь
(дайджест EP-07, ранковий звіт наступного дня).

## Скоуп

- **Детектор — чистий Python** (нуль LLM): після кожної нової активності
  (хук у ST-04-вотчер) перерахувати категорії:
  - найшвидший біг на ~5К / ~10К / ~півмарафон (по avg pace серед активностей
    відповідної дистанції ±5%);
  - найдовша дистанція / найтриваліший біг;
  - найбільший тиждень/місяць за км;
  - all-time high VO2max; найкращий race prediction на кожну дистанцію
    (з `extra`, тригер — щоденний фетч).
- Таблиця `personal_records` (user_id, kind, value, activity_id?, date,
  previous_value) — історія рекордів, не тільки поточний.
- **Telegram**: окреме повідомлення «🎉 Новий рекорд: …» одразу після
  автоаналізу пробіжки (або вбудувати рядком у нього — дешевше і не спамить).
- `/records` — команда з бестами; блок рекордів місяця в дайджесті EP-07;
  сторінка/блок у веб-дашборді EP-04.
- Cold start: перший прогін по всій історії позначає рекорди «заднім числом»
  без повідомлень (тільки заповнення таблиці).

## Acceptance criteria

- [x] Жодних привітань під час backfill/import — тільки для свіжих активностей.
      (`announce_worthy` — дата-гейт `FRESH_DAYS`; CLI `backfill-records` тихий.)
- [x] Дистанційні бести стійкі до сміття: мінімальна дистанція для категорії
      (±5% вікно), біг тільки (`type like %run%`), нереальний темп (<2:30/км) відкидається.
- [x] Рекорд згадується в найближчому ранковому звіті (в контекст `run_analysis`,
      і так — у cache key). Плюс блок у тижневому дайджесті.
- [x] `/records` працює і без жодного рекорду (порожній стан з поясненням).

**Реалізовано** (2026-07): `app/records.py` (детектор) + `PersonalRecord` +
`repository.current_records`/`recent_records`; хук `_records_check_for_user` у morning-тіку,
`/records`-команда, `records` у `run_analysis`/`run_digest` (+ cache key), CLI `backfill-records`;
`tests/test_records.py`. **Не в скоупі v1** (за тікетом): «найшвидші 5К всередині довшого бігу»
(потрібні повні серії), веб-блок (EP-04).

## Підводні камені

- «Найшвидші 5К в межах довшого бігу» (як у Strava) вимагали б повних серій —
  наш `series` downsampled до ~150 точок; v1 свідомо рахує тільки по цілих
  активностях, це чесно задокументувати в `/records`.
- Race predictions коливаються щодня — рекордом вважати покращення понад поріг
  (напр. ≥10 c), інакше спам.
