# ST-16 · Повторне підтягування series/exercise sets в обхід дискового кешу

**Тип:** покращення (BA-аналіз 2026-07-23, кат. A — керування даними) · **Оцінка:** S ·
**Пріоритет:** середній · **Поверхня:** обидва · **Залежності:** ST-15 (ресинк-ендпоінт)

## Проблема

Дисковий кеш вважає series/splits/exercise sets **immutable** (`series:v2:<id>`,
`splits:v1:<id>`, `exercise:v2:<id>` — TTL 365 днів у `app/garmin/client.py`). Але
перший фетч буває передчасним: активність щойно синкнулась, Garmin ще не дорахував
деталі — і порожня/куца відповідь застигає в кеші на рік. EP-15 обійшов це один раз
глобально (бампом `series:v1`→`v2`), користувач так не може: єдиний спосіб — ssh і
`rm garmin_cache/series:v2:<id>.json`. ST-15 без цього тікета ресинкне активність,
але series повернуться з того самого битого кешу.

## User story

Як користувач я хочу прапорець «ігнорувати кеш» при ресинку активності, щоб битий
або неповний закешований графік темпу/HR чи список вправ підтягнувся з Garmin заново.

## Обсяг

**Входить:** чекбокс «в обхід кешу» на веб-кнопці ресинку (ST-15); `/resync <id> force`
у боті; параметр `force=True` у фетчерах series/splits/exercises (пропустити read,
записати свіжий результат назад у кеш); helper видалення одного ключа.

**Не входить:** масова інвалідація кешу (ST-20); обхід кешу `workout:v2` (7-денний
TTL сам розрулює).

## Acceptance criteria

- [ ] `fetch_activity_series`/`fetch_activity_splits`/`fetch_exercise_summary`
      приймають `force: bool = False`; при `True` дисковий read пропускається,
      свіжа відповідь **перезаписує** файл кешу (наступні читачі бачать нове).
- [ ] Порожня відповідь Garmin при `force` НЕ перезаписує наявний непорожній кеш
      (захист від «затерли добре поганим») — логується і повертається старе.
- [ ] Веб/бот-шлях ST-15 пробрасывает прапорець наскрізь.
- [ ] Тести: force оминає кеш, перезаписує файл, порожнє-не-затирає.

## Технічні дотики

- `app/garmin/client.py`: `force`-параметр у трьох фетчерів + `cache_del(key)`;
  in-process мемо теж інвалідувати.
- `app/routers/me.py` / `bot/handlers.py`: прапорець поверх ST-15.

## Вартість LLM / запити до Garmin

LLM: 0. Garmin: +2–3 запити на активність, лише за явною дією.

## Ризики

- Юзер тисне force багато разів поспіль → rate limiter вже кидає паузи; можна
  додати guard «не частіше 1/хв на активність» через in-process мемо.

## RICE

Reach 2/міс · Impact 2 · Confidence 0.8 · Effort ~2 дні → **Score ≈ 1.6**
