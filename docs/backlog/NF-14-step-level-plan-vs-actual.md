# NF-14 · План/факт на рівні кроків: «чи вибіг ти інтервали»

**Тип:** нова фіча (BA-аналіз 2026-07-21, друга партія) · **Оцінка:** M ·
**Пріоритет:** високий (унікальне замикання петлі — рівень Runna/платного тренера;
головний вхід для чесної адаптації) · **Залежності:** EP-01 ✅ (матчинг сесії),
push структурованих `steps` на годинник ✅

## Суть

Петля план/факт замкнена лише на рівні *сесії* (EP-01: «темповик виконано»). Але ми
пушимо на годинник **структуровані** тренування з pace-діапазонами по кроках
(`PlannedWorkout.steps` → Garmin DTO) — а чи влучив користувач у ці діапазони, ніхто
не перевіряє. «Виконав інтервали» може означати «пробіг 8×400 у цілі» або «здувся
після третього» — для адаптації це протилежні сигнали.

## Дизайн

- `client.fetch_activity_splits(activity_id)` — lap/splits активності
  (`/activity-service/activity/{id}/splits`, той самий connectapi-шар; кандидат на
  disk-кеш як immutable asset, ключ `splits:v1:<id>`).
- `app/stepmatch.py` — чистий Python (нуль LLM): розгорнути планові `steps`
  (рекурсивні `repeat` → плоский список робочих відрізків з `pace_min_km
  [fast, slow]` і `dist_m`/`dur_s`), зіставити з фактичними lap'ами (структуроване
  тренування на годиннику даэ lap на крок — порядкове зіставлення з допуском по
  дистанції/тривалості), для кожного робочого кроку — hit/miss по темпу →
  `{steps_hit, steps_total, misses:[{step, planned, actual}]}`.
- Зберігати на `ActivityRecord` (нове JSON-поле `step_match`, поруч із
  `subjective`; міграція). Рахувати в morning-тику одразу після автоаналізу для
  активності, зматченої (EP-01) з сесією, що має `steps`.
- Споживачі: (1) `activity_payload` → `SYSTEM_ACTIVITY` бачить по-кроковий
  результат (**і `_activity_cache_key`** — наскрізна пастка); (2) бейдж
  «🎯 7/8 у цілі» в DM автоаналізу та на `/me`-детейлі; (3) EP-02-адаптація:
  компактний агрегат «останні N ключових сесій — частка кроків у цілі» в контекст
  `run_plan_adaptation` («систематично не вивозить заданий темп» ≠ «пропустив»).

## Acceptance criteria

- [ ] `stepmatch.flatten_steps` / `stepmatch.match(steps, laps)` покриті тестами:
      рівні інтервали в цілі; здувся на останніх; lap'ів менше, ніж кроків
      (зупинив раніше) — чесний partial; вільний біг без структури → `None`;
      warmup/recovery-кроки не рахуються як «робочі» промахи.
- [ ] Активність без матчу/без `steps`/без splits → тихий скіп (best-effort, як
      увесь watch-шлях); фейл фетчу splits ніколи не ламає автоаналіз.
- [ ] `step_match` у `SYSTEM_ACTIVITY`-контексті **і** в `_activity_cache_key`.
- [ ] Бейдж у DM автоаналізу і на `/me`-детейлі активності.
- [ ] Агрегат у контексті `run_plan_adaptation` (не кешується — ок).

## Підводні камені

- Порядкове зіставлення ламається, якщо користувач пробіг НЕ запушене тренування
  (ручні lap'и кнопкою): гейтити фічу на «активність зматчена з сесією, яку ми
  пушили» (`garmin_workout_id` на сесії) — інакше `None`, без вгадувань.
- Пейс у lap'ах — середній по колу: короткі відрізки (<200 м) шумні; допуск
  (напр. ±5%/±3 с/км поза діапазоном — усе ще hit) винести в константи модуля.
- Не перетворити бейдж на батіг: промпти (`SYSTEM_ACTIVITY`/адаптація) читають
  misses як сигнал калібрування темпу плану, не як «погано старався» — тон
  фіксується в промпті одразу.

## Вартість

Нуль нових Claude-викликів (їде в наявному activity-аналізі й адаптації);
+1 Garmin-фетч на пробіжку зі структурою (під наявним rate limiter'ом).
