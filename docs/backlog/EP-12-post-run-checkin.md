# EP-12 · Пост-тренувальний check-in (RPE + самопочуття)

**Тип:** епік · **Оцінка:** M–L · **Пріоритет:** високий (дешевий, а суб'єктивка —
те, чого Garmin не має) · **Залежності:** ST-04 ✅ (автоаналіз після синку — точка входу)

**Перетини (ANALYSIS.md §1.6):** болі/RPE живлять детектор EP-08 і NF-04 (травматичний радар) — тримати схему відповідей придатною для правил.

## Сторя

> Як бігун, я хочу після кожного тренування відповісти на два тапи в Telegram —
> «наскільки важко було? (RPE 1–10)» і «щось боліло?» — щоб тренер знав не тільки
> що показав годинник, а і як воно **відчувалось**, і адаптував план з огляду на це.

## Контекст

Уся аналітика зараз — об'єктивні метрики. Але класичне тренерське правило:
розрив між об'єктивним навантаженням і суб'єктивним відчуттям — найраніший
сигнал перевтоми/хвороби/травми, що насувається. ST-04 уже шле повідомлення
після синку нової пробіжки — check-in природно чіпляється до нього.

## Скоуп

- Після автоаналізу пробіжки — inline-клавіатура RPE (1–10, один тап) +
  опційна кнопка «🩹 щось боліло» → уточнення текстом (нога/коліно/…).
- Зберігання: `ActivityRecord.subjective` (JSON: `{rpe, pain?, note?}`) —
  callback пише по activity id; повторний тап перезаписує.
- **Споживачі** (по одному на фазу):
  1. `/activity`-аналіз бачить RPE («темп легкий, але RPE 8 — це дзвіночок»);
  2. адаптивна джоба (EP-02) отримує RPE-тренд + болі в контекст;
  3. matching EP-01: сесія «виконана, але RPE значно вище очікуваного» —
     окремий статус для тижневого дайджесту (EP-07).
- `/checkin` — ручна команда для останньої пробіжки (якщо проігнорив кнопки).
- Болі: якщо той самий біль повторюється ≥2 рази за 14 днів — підняти в
  ранковий звіт з порадою полегшити/до лікаря (синергія з EP-08 алертами).

## Acceptance criteria

- [x] Кнопки не спамлять: один check-in на активність, тихо зникають після
      відповіді (edit_message), TTL — до наступної пробіжки.
- [x] RPE-дані входять у cache key усіх промптів, куди потрапляють (наскрізна
      пастка з README).
- [x] Мовчання — валідна відповідь: жодних нагадувань, аналітика працює й без RPE.
- [x] `/me` показує RPE у списку активностей (одна колонка).

## Статус — MVP done (2026-07)

Реалізовано ядро + споживач-фаза 1:

- `ActivityRecord.subjective` (JSON `{rpe, pain?, note?}`, міграція
  `c1d2e3f4a5b6`) + `repository.set_subjective` (мердж, ре-тап перезаписує) /
  `get_last_activity`.
- Бот: `checkin_keyboard` (RPE 1–10 + «🩹 щось боліло») і `checkin_callback`
  (pattern `^ci:`) — **stateless**, activity id зашитий у callback data (пастка з
  тікета). Біль уточнюється кнопками частин тіла (коліно/гомілка/…), а не вільним
  текстом, тому стан не потрібен. Клавіатура чіпляється до автоаналізу
  (`bot/jobs.py`) і до `/activity`; ручна `/checkin [rpe] [note]` — для останньої
  пробіжки.
- Споживач-фаза 1: `activity_payload` віддає `subjective` у `SYSTEM_ACTIVITY`
  (аналітик підсвічує розрив об'єктив/суб'єктив і біль) — і воно автоматично в
  dedup-cache key (`_activity_cache_key` хешує payload).
- `/me`: RPE + біль у картці активності та на детальній сторінці.
- `tests/test_checkin.py`.

## Статус — фази 2–3 done (2026-07)

Усі три споживачі суб'єктивки зроблено через новий чистий (нуль LLM) модуль
`app/subjective.py` (агрегатор над `repository.recent_subjective_runs`, сусід
`injury.py`, але лише *формує* сигнал, не скорить його). `subjective.summarize(runs)`
→ `{n, avg_rpe, rpe_rising, recurring_pain?, recent}`:

- **Фаза 2 (EP-02 адаптація):** `run_plan_adaptation` кладе `subjective` у контекст,
  `SYSTEM_PLAN_ADAPT` полегшує найближчі важкі сесії, коли `rpe_rising` або
  `recurring_pain`, навіть при нормальному об'єктивному навантаженні. Не кешується.
- **Фаза 3a (дайджест EP-07):** `repository.weekly_compliance` рахує `overreached` —
  легкі за задумом сесії (easy/recovery/base/long), виконані, але з RPE ≥
  `subjective.HARD_RPE` (8): «зробив, але далося важче, ніж мала б ця сесія». Їде
  всередині `compliance` (вже в `_digest_cache_key`); `SYSTEM_DIGEST` читає це як сигнал
  недовідновлення.
- **Фаза 3b (ранковий звіт):** `run_analysis` → `analyze_with_stats`
  `user_content["subjective"]` **і `_cache_key`** (наскрізна пастка); секція
  «САМОПОЧУТТЯ / ЧЕК-ІНИ» у `SYSTEM` підсвічує повторюваний біль / зростання зусилля.
  Це контекст у самому звіті — комплементарно до окремого NF-04 push-адвайзорі (той
  шле рідкісне попередження, цей — визнає стан у щоденному тексті).

`rpe_rising` реюзає той самий pace-stable/RPE-up тест, що `injury._rpe_signal`;
`recurring_pain` — те саме правило ≥2×/14 днів. `tests/test_subjective.py`.

## Підводні камені

- Callback-и бота зараз тримають стан у `context.user_data` — для check-in стан
  не потрібен (activity id зашитий у callback data), не ускладнювати.
- Не перетворити на анкету: два тапи максимум, інакше юзер перестане відповідати
  за тиждень.
