# EP-11 · Веб-чат з тренером

**Тип:** епік · **Оцінка:** L · **Пріоритет:** середній (логічний після EP-04
дашборда; сильно виграє від EP-09) · **Залежності:** EP-09 бажано (той самий движок)

## Сторя

> Як користувач за ноутбуком, я хочу чат із тренером прямо у вебі — з тією ж
> пам'яттю і даними, що в Telegram: спитати про сон, попросити змінити план
> (з тим самим ✅/🛡/❌ підтвердженням), розібрати пробіжку — не перемикаючись
> у телефон.

## Контекст

Уся «розумна» логіка вже канало-незалежна (`run_ask`, `run_plan_edit`,
`run_activity_analysis` приймають session+user), Telegram-специфіка — тільки
UI-обгортка в `bot/handlers.py` (кнопки, `context.user_data["pending_plan"]`).
Веб має логін-сесії і templates. Бракує: сторінки чату, стрімінгу і сховища
pending-станів поза telegram-контекстом.

## Скоуп

- `GET /chat` (login) — одна сторінка: історія + інпут; vanilla JS у стилі
  проєкту (як hover у `detail.html` — прогресивне покращення, без фронт-фреймворка).
- `POST /chat` → SSE-стрімінг відповіді (`text/event-stream`; Anthropic streaming
  API — привід перейти на `AsyncAnthropic`, синергія з PERF-04).
- Роутер повідомлень: питання → `run_ask`-движок (з EP-09 — з інструментами);
  «зміни план …» → `run_plan_edit` з тими ж trybами підтвердження, тільки
  кнопки — HTML (`pending_plan` переїжджає з `context.user_data` у БД/таблицю,
  щоб працювало в обох каналах і переживало рестарт — досі відомий недолік бота).
- Історія чату — нова таблиця (`chat_messages`) або розширений `ReportLog`;
  `get_recent_asks`-тред читає з того ж місця для обох каналів.

## Acceptance criteria

- [ ] Чат бачить ті самі дані/план, що бот; план-едит із вебу відображається
      в боті і навпаки (спільний pending-стан у БД).
- [ ] Стрімінг: перша літера відповіді < ~2с (не чекати повної генерації).
- [ ] Кожен виклик — у `ReportLog` як завжди (вартість під контролем).
- [ ] Без JS сторінка деградує до form-submit + повна відповідь (як решта вебу).
- [ ] Мобільна верстка стерпна (це ж особистий інструмент, не продукт).

## Підводні камені

- Перенесення `pending_plan` у БД зачіпає бот-флоу (`plan_callback`) — зробити
  окремим підготовчим PR, він цінний сам по собі (pending переживає рестарт бота).
- SSE + cookie-сесія: перевірити, що SessionMiddleware не буферизує стрім
  (можливо, віддавати стрім поза middleware-шляхом).
