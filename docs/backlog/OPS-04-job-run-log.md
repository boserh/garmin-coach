# OPS-04 · Журнал фонових джобів (job_runs)

**Тип:** спостережуваність (BA-аналіз 2026-07-23, кат. B) · **Оцінка:** M ·
**Пріоритет:** високий · **Поверхня:** веб (`/me/jobs` + `/admin/jobs`); бот побічно ·
**Залежності:** — (OPS-05/OPS-07 читають цей журнал, але вміють і без нього)

## Проблема

Шість фонових джоб (`morning_job` кожні 20 хв у вікні 07–23 + пʼять `run_daily`:
plan_sync, weather_plan, plan_adapt, weekly_digest, sleep_nudge — `bot/main.py`)
відпрацьовують мовчки: результат видно лише в `bot.log` на Pi, а причини скіпів —
взагалі лише під `LOG_LEVEL=DEBUG`. Питання «чи запускався вчора plan_sync і чим
скінчився?» вимагає ssh + grep. Для системи, яка сама шле повідомлення й пушить
календар, це головна діра довіри: коли щось не прийшло — незрозуміло, чи не мало
прийти, чи впало.

## User story

Як користувач я хочу бачити історію запусків фонових задач по собі (коли, що,
результат, причина скіпу/помилки), щоб за хвилину зрозуміти, чому щось не прийшло.

## Обсяг

**Входить:** таблиця `job_runs` (id, job, user_id nullable, started_at, finished_at,
status ok/skip/error, detail — «sent» / «outside window» / «MFARequired» / хвіст
трейсбека ≤512 симв.; міграція); запис із `for_each_user`-обгортки + верхнього рівня
джоб; ротація 30 днів (лінивий purge на write, як llm_cache); `GET /me/jobs` (свої
запуски) і `GET /admin/jobs` (всі); рядок `last_morning_status` у `/status`.

**Не входить:** перезапуск джоб з UI (є `/test_morning`, `/deploy`); метрики поза
запусками (тривалість — так, бо є started/finished; пам'ять/CPU — ні).

## Acceptance criteria

- [ ] Кожен запуск per-user гілки джоби лишає рівно один рядок зі статусом і
      причиною; помилка одного юзера не губить рядки інших (наявний try/except у
      `for_each_user` — точка запису).
- [ ] 20-хв тік НЕ засмічує журнал: рутинні скіпи («guard: уже надіслано»,
      «поза вікном») агрегуються — один рядок на юзера/день з лічильником тіків,
      а нетипові скіпи й помилки пишуться окремо.
- [ ] `/me/jobs` показує останні 50 своїх запусків; `/admin/jobs` — фільтр по job.
- [ ] Ротація: рядки старші 30 днів зникають без ручного втручання.
- [ ] Тести: запис ok/skip/error, агрегація тіків, ізоляція по юзеру, ротація.

## Технічні дотики

- `app/db/models.py` (+Alembic): `JobRun`.
- `bot/jobs.py`: `for_each_user` + `_tick_for_user`/`_sync_for_user`/… — обгортка
  `record_job_run(...)`; `bot/main.py` без змін (обгортка всередині).
- `app/routers/me.py`, `app/routers/admin.py` (реюз table.html-рендера),
  `app/routers/health.py::status`.

## Вартість LLM / запити до Garmin

0 / 0. Пише кілька коротких рядків у SQLite на джобу — на Pi невідчутно (WAL уже
ввімкнений для конкурентних bot/web-процесів).

## Ризики

- Розростання таблиці — ротація в AC. Подвійний запис при ретраї PTB — ідемпотентний
  ключ не потрібен, дубль-рядок ок (це журнал).
