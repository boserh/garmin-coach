# EP-08 · Проактивні health-алерти (аномалії відновлення)

**Тип:** епік · **Оцінка:** L · **Пріоритет:** середній-високий (унікальна цінність:
годинник це знає, але мовчить) · **Залежності:** немає (дані вже збираються)

**Перетини (ANALYSIS.md §1.6):** EP-12 (болі) та NF-04 — один детекторний шар «сигнали ризику», різні джерела правил; базлайни NF-01 дають кращі пороги, ніж хардкод.

## Сторя

> Як користувач, я хочу отримати повідомлення, коли метрики тихо повзуть у поганий
> бік — HRV нижче baseline кілька днів поспіль, RHR росте, сон системно короткий,
> ACWR у червоній зоні — з поясненням і порадою (полегшити тиждень / можливо,
> захворюю) — не чекаючи, поки я сам це помічу в ранковому звіті.

## Контекст

Ранковий звіт бачить зріз «сьогодні», але ніхто не дивиться на **тренд як тригер**:
`daily_metrics.extra` вже містить resting_hr, hrv baseline band, sleep need,
readiness, acwr_pct — історія в БД, `repository.read_daily_metrics` user-scoped.
Морнінг-тік уже проходить по юзерах щодня — детектор чіпляється до нього
безкоштовно (без нового Garmin-фетчу).

## Скоуп

- **Детектор — чистий Python, без LLM**: правила на кшталт «HRV < нижньої межі
  baseline ≥3 дні», «RHR +5 уд до 7-денної медіани», «ACWR > 1.5», «сон < потреби
  ≥4 з 7 днів». Кожне правило → типізований `Alert(kind, severity, evidence)`.
- **LLM тільки для тексту**: якщо детектор щось знайшов — один Sonnet-виклик
  перетворює evidence на людське повідомлення (або навіть без LLM у v1 —
  шаблонні тексти, LLM додати потім).
- Cooldown per-rule у `bot_state` (`alert:<kind>` — не частіше раза на N днів),
  щоб не спамити тим самим трендом щодня.
- Опційний per-user вимикач у `/settings` (`alerts_enabled`, дефолт on).
- Синергія: активний алерт передається в ранковий звіт як контекст (той сам
  згадає «HRV третій день низько») — але push окремим повідомленням, бо тренд
  може зламатись і ввечері (після вечірнього синку).

## Acceptance criteria

- [x] Правила задокументовані в коді поряд із порогами; пороги в константах
      (`app/health.py` — `SUSTAIN_DAYS`/`SLEEP_DAYS`/`RECENT_WINDOW`/… + порогом служить
      особистий базлайн-коридор NF-01, не хардкод).
- [x] Жодного алерта на юзера з <7 днями історії (`HEALTH_MIN_HISTORY_DAYS` cold-start
      gate → `level="calibrating"`; метриці ще й треба 14 днів на band, тож старт тихий).
- [x] Cooldown працює (per-rule `bot_state` `alert:<kind>`, `HEALTH_ALERT_COOLDOWN_DAYS`);
      деактивований юзер / вимкнені алерти (`User.alerts_enabled` + `HEALTH_ALERTS`) — тиша.
- [x] Тести детектора на синтетичних рядах (`tests/test_health.py`).
- [x] Не медичні діагнози (`SYSTEM_HEALTH` + `health.summary` — «схоже на недовідновлення»
      + порада до лікаря при тривожних патернах, жодних діагнозів).

## Статус — ✅ зроблено (2026-07)

- **Детектор** `app/health.py` (чистий, нуль LLM): реюзає особисті персентиль-коридори
  NF-01 (`baselines.compute_baselines`) як пороги й ловить метрику, що кілька днів тримається
  ПОЗА коридором у поганий бік — `hrv_low`, `rhr_up`, `sleep_debt`, `stress_high`. Кожне →
  `Alert(kind, severity, detail, advice)`; `detect(history)` →
  `HealthReport(level calibrating/none/alert)`.
- **Сервіс:** `service.build_health_alerts` (спільний для `/health` і джоби),
  `service.run_health_alert` (наратив через Sonnet `SYSTEM_HEALTH` з детермінованим
  fallback `health.summary`), `ReportLog(kind="health")`, `MODEL_HEALTH`.
- **Хук** `_health_check_for_user` у morning-тіку (після injury): DM на КОЖЕН свіжий вид
  алерту, per-rule cooldown `alert:<kind>`; **скіп, якщо injury-адвайзорі вже пішов сьогодні**
  (максимум один risk-DM/день). Команда **`/health`** — миттєвий DB-рід.
- **Тумблери:** процесний `HEALTH_ALERTS` + per-user `User.alerts_enabled` (форма
  `/settings`, дефолт on; міграція `b1c2d3e4f5a6`).
- **НЕ зроблено (свідомо):** активний алерт як контекст у ранковому звіті —
  звіт уже отримує стан відновлення через NF-01 `norm`; ця синергія лишена future
  extension. Ідея «тихого запуску» (лог замість повідомлення N тижнів) досяжна через
  `HEALTH_ALERTS=false`.

## Підводні камені

- False positives вбивають довіру швидше, ніж false negatives — пороги
  консервативні, severity потрібен, і v1 краще запустити «тихо» (лог замість
  повідомлення) на 2–3 тижні для калібрування на реальних даних.
