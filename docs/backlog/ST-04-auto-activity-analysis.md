# ST-04 · Автоаналіз нової пробіжки після синку

**Тип:** покращення · **Оцінка:** M · **Пріоритет:** високий (сильний product-ефект) ·
**Залежності:** немає

## Сторя

> Як бігун, я хочу отримувати розбір пробіжки в Telegram автоматично, щойно вона
> синкнулась із годинника — без ручного `/activities` → `/activity <id>`.

## Контекст

`run_activity_analysis` повністю готовий: будує компактний payload із сегментами
темпу/HR, кличе Sonnet із `SYSTEM_ACTIVITY`, пише текст на `ActivityRecord.analysis`
і логує `ReportLog(kind="activity")`. Бракує лише тригера «з'явилась нова активність».
Природне місце — цикл `morning_job`-а, який і так ходить по користувачах кожні 20 хв
і кличе `build_payload_cached` → `persist_payload`.

## Acceptance criteria

- [ ] Нова running-активність, що з'явилась при синку, отримує аналіз і він
      надсилається в чат користувача (одним повідомленням, з назвою/датою/дистанцією).
- [ ] Кожна активність аналізується **рівно один раз** (ре-старти бота, повторні
      синки того ж дня — без дублів).
- [ ] Не-running активності: вирішити — або теж аналізувати (механіка підтримує),
      або слати тільки для бігу; за замовчуванням — тільки біг.
- [ ] Backfill-активності (import-export, давніші за N днів) НЕ тригерять розсилку.
- [ ] Помилка Claude/Telegram не ламає morning-цикл (best-effort, як plan_sync).
- [ ] Вартість під контролем: аналіз лише для активностей свіжіших за ~2 дні.

## План по файлах

- `app/garmin/repository.py` —
  - `upsert_activity` (~:212): повертати прапорець `inserted` (нова vs існуюча);
  - `persist_payload` (~:243): збирати і повертати список **нових** activity-рядків
    (id + type + start date) — сигнатуру розширити акуратно, виклики оновити.
- `bot/jobs.py` — у циклі по користувачах (поряд із `_morning_for_user`, той самий
  20-хв тік, але БЕЗ вікна 07–12 і once-a-day guard): для нових run-активностей
  свіжіших за 2 дні → `run_activity_analysis` → надіслати текст у `telegram_chat_id`.
  Захист від повторів двошаровий: (1) «нова» = щойно вставлена в цьому тіку,
  (2) `ActivityRecord.analysis is None` — природний маркер (аналіз пишеться на рядок).
- `bot/main.py` — якщо винести в окремий джоб `activity_watch_job` (чистіше, бо
  morning_job — інша відповідальність, як з plan_sync_job): зареєструвати через
  `JobQueue.run_repeating`. Рішення на імплементації; окремий джоб преферований.
- `app/analysis/service.py` — без змін (`run_activity_analysis` ~:487 готовий).
- `tests/` — тест на `persist_payload` (нові vs існуючі); тест guard-ів (уже
  проаналізована / стара / не-run — пропускається).

## Підводні камені

- **CLI-бекфіли** (`import-export`, `backfill-series`) теж кличуть upsert-и — фільтр
  за свіжістю дати обов'язковий, інакше імпорт GDPR-експорту вистрелить сотнею аналізів.
- Окремий джоб означає ще один `build_payload_cached`-виклик — НЕ робити цього:
  або вбудуватись у наявний цикл morning_job (він уже фетчить), або джоб має
  працювати від щойно-персистнутих даних без повторного фетчу Garmin.
- `/activity <id>` вручну на вже проаналізованій активності — дедуп-кеш і так
  віддасть той самий текст; конфлікту немає.
