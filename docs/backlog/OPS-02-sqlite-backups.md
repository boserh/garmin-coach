# OPS-02 · Бекапи `garmin.db` (SQLite на SD-картці Pi — один файл тримає все)

**Тип:** операційна стійкість · **Оцінка:** S · **Пріоритет:** високий за
співвідношенням ціна/ризик (скрипт на вечір проти втрати всієї історії) ·
**Залежності:** — (деплой на Pi; локально — лише написати й протестувати скрипт)

## Проблема

Уся цінність системи — річна історія `daily_metrics`, серії пробіжок, плани,
`report_logs` з вартістю — живе в **одному файлі** `garmin.db` на SD-картці
Raspberry Pi. SD-корупція — найтиповіший режим відмови Pi-деплою; друге за
ймовірністю — невдала міграція (alembic `upgrade` на живій базі без відкату).
Бекапів немає взагалі: ні планових, ні перед міграціями. `git` історію коду
захищає, дані — ніщо.

Fernet-шифровані креденшели в базі без `APP_SECRET_KEY` нечитабельні — тобто
бекап самої БД **безпечно** зберігати поруч; ключ (`.env`) бекапиться окремим
каналом, інакше відновлений бекап не розшифрує креденшели.

## Acceptance criteria

- [ ] `scripts/backup_db.py` (або чистий sqlite3-однорядник у shell): онлайн-бекап
      через `VACUUM INTO` / backup API — **не** `cp` (бот і веб пишуть у файл;
      копія на живій базі може бути порваною). Виходить
      `backups/garmin-YYYY-MM-DD.db` + ротація: 7 щоденних, 4 тижневих.
- [ ] systemd timer (або cron) на Pi — щоніч; юніт-файл поруч з майбутніми
      юнітами бота/uvicorn (TODO в CLAUDE.md вже передбачає systemd).
- [ ] Копія поза SD-карткою: хоча б `rsync` останнього бекапу на інший хост /
      USB / хмару — SD-корупція вбиває і базу, і «бекапи» поруч з нею.
- [ ] Перед-міграційний бекап: рядок у CLAUDE.md («перед `alembic upgrade head`
      на Pi — зроби бекап») + виклик скрипта; за бажання — обгортка
      `scripts/migrate.sh`, що робить бекап і тільки тоді мігрує.
- [ ] Документований і **один раз реально виконаний** restore-тест: підняти
      веб на копії бекапу, побачити свої дані (перевірка «бекап читабельний»,
      без якої все інше — театр).
- [ ] `.env`/`APP_SECRET_KEY`: окремий захищений бекап (менеджер паролів /
      зашифрований файл) — задокументувати в тікеті/CLAUDE.md.

## Підводні камені

- Живі проби — на Pi, не на Mac (локально лише написати скрипт і прогнати на
  копії/скретч-базі; ніяких ad-hoc записів у робочу `garmin.db`).
- `VACUUM INTO` вимагає SQLite ≥ 3.27 — на Pi OS це давно так, але перевірити;
  запасний шлях — `sqlite3 garmin.db ".backup '...'"` (онлайн-safe так само).
- `DATABASE_URL` може вказувати не на дефолтний шлях — скрипт має читати шлях
  з `Settings`, а не хардкодити `./garmin.db`.
- Якщо колись переїзд на Postgres (PERF-03) — цей тікет заміниться `pg_dump`;
  не інвестувати в SQLite-специфіку більше за потрібне.
