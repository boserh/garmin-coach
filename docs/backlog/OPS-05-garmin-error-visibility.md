# OPS-05 · Видимість помилок Garmin API (лічильники + останні збої)

**Тип:** спостережуваність (BA-аналіз 2026-07-23, кат. B) · **Оцінка:** M ·
**Пріоритет:** високий · **Поверхня:** обидва (веб /status + банер; бот — разовий DM
при сплеску) · **Залежності:** — (синергія з OPS-04/OPS-07)

## Проблема

`client._safe` ковтає будь-який збій ендпоінта в `{"_error": ...}` і рядок логу;
`_api` після вичерпання 429-ретраїв просто пропагує. OPS-01 навмисно тримає
grep-стабільні маркери (`GARMIN AUTH FAIL`) — але **в лог-файлі на Pi**. Користувач
бачить наслідок (дірки в даних, «неповний» день), а не причину: 403-деградацію
garth, серію 429, Cloudflare-зміну. `/status` робить live-логін (одна точка), але
нічого не каже про те, які ендпоінти падали за останню добу. Для проєкту, чий
головний екзистенційний ризик — саме неофіційний API, це сліпа зона №1.

## User story

Як користувач я хочу бачити на /status і дашборді, чи були за останню добу помилки
Garmin і які, щоб відрізняти «годинник не синкнувся» від «API деградує» — і дізнатись
про деградацію раніше, ніж посиплються звіти.

## Обсяг

**Входить:**
- In-process кільцевий буфер у `client` (останні ~50: ts, суфікс ендпоінта, тип/HTTP-
  статус, після ретраїв) — sync-код без доступу до сесії БД, тому буфер модульний;
  `build_payload_cached`/`resync_*` після фетч-фази зливають його у `bot_state`
  (ключ `garmin_errors`, JSON, обрізаний до 20 останніх + лічильники за 24h).
- `/status`: `garmin_errors_24h` (кількість, розбивка 401/403/429/5xx/інше) +
  останній збій; банер на `/dashboard` при >0 за 24h.
- Бот: разовий DM «⚠️ Garmin API: N збоїв за годину (типи)» при сплеску
  (>`GARMIN_ERROR_BURST`=10/год), guard раз/добу через `bot_state`.
- Окремий випадок **auth-збою** (`GARMIN AUTH FAIL`): DM одразу (раз/добу) — це
  тригер міграції з OPS-01, він не має чекати грепа логів.

**Не входить:** автоматичні дії (перемикання провайдера — OPS-01 свідомо ручний);
історія довша за 24–48h (для тренду є логи).

## Acceptance criteria

- [ ] Збій будь-якого connectapi-виклику потрапляє в буфер із суфіксом ендпоінта і
      класифікацією (401/403/429/5xx/network); одиночний 429, знятий ретраєм, НЕ
      рахується збоєм.
- [ ] `/status` віддає лічильники за 24h + останній збій; дашборд показує банер.
- [ ] Сплеск → один DM на добу; auth-fail → один DM на добу, текст із відсилкою до
      /settings і OPS-01.
- [ ] Нуль нових запитів до Garmin (лише спостереження наявних).
- [ ] Тести: класифікація, зливання буфера, guard DM-ів.

## Технічні дотики

- `app/garmin/client.py`: буфер + hook у `_safe`/`_api` (після ретраїв);
  `recent_errors()`/`drain_errors()`.
- `app/garmin/service.py` (злив у bot_state), `app/routers/health.py::status`,
  `app/routers/dashboard.py` (банер), `bot/jobs.py::_tick_for_user` (DM-гейти).

## Ризики

- Буфер — per-process (bot і web мають свої): зливати з обох шляхів у спільний
  `bot_state`; не «чинити» це глобальним стейтом (та сама філософія, що
  ratelimit SEC-01).
- Шум від очікуваних 403 (та сама RHR-історія) — класифікувати очікувані ендпоінти
  окремо або виключити зі сплеск-лічильника.
