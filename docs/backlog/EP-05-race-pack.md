# EP-05 · Race pack — підготовка до перегонів

**Тип:** епік · **Оцінка:** L · **Пріоритет:** середній (цінність пікова перед стартом) ·
**Залежності:** фаза 0 (типізований `target_date` в intake) — блокер усього іншого;
погодний хелпер ділити з EP-13 (`fetch_forecast_week`); фаза 2 ділить GAP-модуль
з EP-15

*(переписано за аудитом 2026-07 — ANALYSIS.md §1.2)*

## Сторя

> Як бігун, що готується до перегонів за планом, я хочу за тиждень до старту
> отримати «race pack»: пейсинг-стратегію по кілометрах (з моїх race predictions),
> план харчування/гелів, поради під погоду і чекліст напередодні — щоб на старт
> вийти з готовим сценарієм.

## Контекст

Усі інгредієнти вже збираються: race predictions (5K/10K/half/marathon у секундах),
endurance score, VO2max — у `DailyMetric.extra`; цільова дата/дистанція — в intake
активного плану; погода — `app/weather.py`; тейпер — частина згенерованого плану.
Бракує синтезу в один документ і тригера за N днів до старту.

## Фази

- **Фаза 0 — блокер, мікро-сторя**: перевірити фактичний ключ цілі у
  `TrainingPlan.params`/intake на реальних даних (ANALYSIS.md §4.4.6) і
  **типізувати `target_date`/`target_dist`**. Без структурованої дати тригер
  «за 7 днів до старту» неможливий — раніше це ховалось у «підводних каменях»,
  але це передумова, з неї починати.
- **Фаза 1 — ядро**: `/race` + автотригер + блок на `/plan` (AC нижче).
- **Фаза 2 — опційно**: сегментний пейсинг по рельєфу траси (розбивка по км
  з поправкою на профіль висот, не тільки тактична). Потребує GAP-функцію —
  писати її **одним реюзабельним модулем** з EP-15, не дублювати.

## Acceptance criteria (фаза 1)

- [ ] `/race` у боті (з активним планом, що має цільову дату/дистанцію) → markdown:
      цільовий і запасний темп (за race prediction ± корекція на погоду/рельєф),
      розбивка по км (перші км обережніше, negative split), харчування (гелі/вода
      по хвилинах для дист. ≥ half), чекліст напередодні, розминка.
- [ ] За 7 днів до цільової дати бот шле race pack сам — рівно один раз
      (guard у `bot_state`).
- [ ] Дата ≤ 7 днів → у pack включено прогноз погоди на день/місце старту
      (якщо у користувача задано location).
- [ ] Блок «Перегони» на `/plan`, коли до цілі ≤ 14 днів (текст останнього pack
      із `report_logs`).
- [ ] Виклик на Opus, логується як `ReportLog(kind="race")`; повторний `/race`
      без нових даних — кеш-хіт.
- [ ] Немає активного плану або цільової дати → людське пояснення, не помилка.

## План по файлах

- Фаза 0: `app/routers/plan.py` (setup-форма) + intake — типізовані
  `target_date`/`target_dist`; за потреби — легка міграція/бекфіл наявного плану.
- `app/analysis/prompts.py` — `SYSTEM_RACE`: вхід — ціль (дата/дистанція/цільовий
  час, якщо заданий в intake), `fitness`-знімок (predictions, endurance, VO2max),
  останні тижні плану (обсяг/довгі), погода (опційно); вихід — структурований
  markdown (звичайний текст — без JSON-парсингу).
- `app/analysis/service.py` — `run_race_plan(session, user_id, api_key)` за
  зразком `run_analysis`: контекст із `repository.get_active_plan` (intake),
  `get_recent_extra`, `weekly_run_volume`; прогноз на день старту — **реюзати
  хелпер тижневого прогнозу з EP-13** (`fetch_forecast_week` в `app/weather.py`),
  не писати свій; Opus (`MODEL_DEEP`); дедуп-ключ включає ціль+знімок+погоду;
  `ReportLog(kind="race")`.
- `bot/handlers.py` — команда `/race`; повідомлення-заглушка без плану/цілі.
- `bot/jobs.py` — тригер у денному джобі (можна в `plan_sync_job` — він уже
  daily): якщо `target_date - today == 7` і активний план → `run_race_plan` →
  надіслати; guard `bot_state` (`race_pack_sent:<plan_id>`).
- `app/routers/plan.py` + `plan.html` — блок «Перегони ≤ 14 дн»: дістати останній
  `report_logs` рядок kind="race" цього користувача, рендер markdown.
- `app/garmin/repository.py` — `get_last_report_of_kind(session, user_id, kind)`
  (узагальнення `get_last_report`).
- `tests/` — тригер рівно за 7 дн і рівно раз; `/race` без плану; включення
  погоди лише в межах 7 дн.

## Підводні камені

- Race predictions — це прогноз Garmin у ідеальних умовах: промпт має явно
  коригувати на погоду/рельєф і давати «цільовий + запасний» темп, а не одну цифру.
- Тейпер уже в згенерованому плані — pack не повинен пропонувати інший тейпер,
  лише посилатись на наявні сесії останнього тижня (передати їх у контекст).
