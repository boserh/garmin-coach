# NF-15 · Пробіг кросівок і нагадування про заміну

**Тип:** нова фіча (BA-аналіз 2026-07-21, друга партія) · **Оцінка:** M (левова
частка — розвідка реального shape gear-endpoint'ів) · **Пріоритет:** низький–середній
(відчутна щоденна користь без LLM; підживлює injury-радар) · **Залежності:** — ;
опційний вхід у NF-04

## Суть

Зношені кросівки — банальний і частий фактор травм, а наш травматичний радар про
взуття нічого не знає. Garmin зберігає gear і прив'язку до активностей, але в наших
даних цього немає. Пропозиція: тягнути gear під час звичайного синку, рахувати
пробіг на пару чистим Python і одноразово нагадувати при порозі зносу.

## Дизайн

- `client.fetch_gear` (`/gear-service/...`; список gear юзера + gear→activity
  прив'язка) — той самий connectapi-патерн, під наявним rate limiter'ом; список
  gear кешувати коротко (7д, як `workout:v2`), прив'язку до активності — як
  immutable (`gear_link:v1:<activity_id>`).
- Зберігання: `ActivityRecord.gear_id` (nullable, міграція) заповнюється при
  фетчі нових активностей; агрегатор `app/gear.py` (нуль LLM) рахує пробіг пари
  = сума `distance_km` її активностей + `initial_km` з Garmin (пара могла жити до
  нас).
- Поріг (`GEAR_WEAR_KM`, дефолт 700; `0` вимикає) → одноразовий DM
  «👟 <пара> набігала ~750 км — подумай про заміну» в morning-тику, guard через
  `bot_state` `gear_warned:<gear_id>` (патерн `injury_warned`; скидається, якщо
  пара знову далеко від порога — тобто ні: gear_id стабільний, один ворнінг на
  пару, повторний — хіба на +150 км понад поріг).
- Команда `/gear`: список пар з пробігом і датою останнього використання
  (чистий DB-рід після першого синку).
- Опційно (окремим кроком): «стара пара (>поріг) + свіжий біль гомілка/коліно
  (EP-12)» — п'ятий сигнал `injury.assess`.

## Acceptance criteria

- [ ] Розвідковий крок зафіксовано в тікеті ДО коду: реальні URL/shape
      gear-endpoint'ів перевірені live (як робилось для strength DTO) — це
      блокер, не деталь.
- [ ] Синк нових активностей заповнює `gear_id` (best-effort: фейл gear-фетчу не
      ламає `build_payload_cached`).
- [ ] `gear.mileage(activities, gear)` покритий тестами: кілька пар, пара без
      активностей, `initial_km`, активності без gear_id.
- [ ] DM один раз на пару при перетині порога; `/gear` показує пробіг; нуль DM
      при `GEAR_WEAR_KM=0`.
- [ ] Нуль Claude-викликів.

## Підводні камені

- Історичні активності лишаться без `gear_id` — бекфіл окремою CLI
  (`backfill-gear --email`, патерн `backfill-series`), інакше пробіг буде
  занижений і ворнінг збрехне в кращий бік; до бекфілу — чесна примітка в `/gear`.
- Дефолтна пара Garmin приписується всім пробіжкам автоматично — якщо юзер не
  веде gear у Connect, цифри будуть фікцією однієї пари; `/gear` має це чесно
  показувати (одна пара на 100% пробіжок → підказка «схоже, gear у Connect не
  ведеться»), а ворнінг — не наполягати.
- GDPR-експорт gear не покриває (або покриває інакше) — import-export-шлях у
  скоуп не тягнути, v1 живе від live-синку + бекфілу по API.

## Вартість

Нуль LLM; +1–2 легкі Garmin-фетчі на синк під наявним rate limiter'ом.
