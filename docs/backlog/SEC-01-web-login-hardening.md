# SEC-01 · Зміцнення веб-логіну: rate limit, секрет сесії, logout через POST

**Тип:** безпека · **Оцінка:** S–M · **Пріоритет:** середній зараз (один-два
юзери за NAT), **блокер перед відкриттям `/register` назовні** (разом з EP-06/
PERF-03) · **Залежності:** —

## Проблема

Три речі, кожна дрібна, разом — відкриті двері, щойно веб дивиться в інтернет:

1. **Немає rate limit на `/login` і `/register`** (`app/routers/auth.py`).
   bcrypt повільний, але перебір нічим не обмежений; `/register` без ліміту —
   спам неапрувнутими акаунтами.
2. **Фолбек-секрет сесії**: без `APP_SECRET_KEY` застосунок тихо стартує з
   `"dev-insecure-session-secret"` (`app/main.py:46`) — будь-хто може **підписати
   собі сесійну куку** з `user_id` адміна. Зараз це «зручний перший запуск»,
   але провал конфігурації виглядає як робоча система.
3. **`/logout` через GET** — стан міняється GET-запитом (сторонній `<img src=/logout>`
   розлогінює). Дрібниця, але безкоштовна до виправлення.

Що вже добре і чого НЕ треба чіпати: `same_site="lax"` закриває крос-сайтові
POST (CSRF-токени для форм не потрібні на цьому етапі); паролі — bcrypt;
креденшели — Fernet.

## Acceptance criteria

- [ ] Rate limit на `POST /login` + `POST /register`: in-memory token-bucket
      per-IP (+ per-email на логін), без нових залежностей — цього досить для
      одного процесу на Pi. Перевищення → 429 з людським текстом на сторінці.
      Ліміти в `Settings` з дефолтами (напр. 5 спроб / 5 хв).
- [ ] Порожній `APP_SECRET_KEY`: **гучний** `logging.error("AUTH: APP_SECRET_KEY
      is not set — sessions are forgeable")` при старті + банер на `/login`
      (як мінімум); розглянути ephemeral-ключ (`Fernet.generate_key()` на процес)
      замість константи — сесії живуть до рестарту, але їх ніхто не підробить.
      Зафіксувати вибір у CLAUDE.md.
- [ ] `/logout` → POST (кнопка-форма в base-шаблоні); старий GET лишити
      редіректом на сторінку з кнопкою або прибрати одразу — узгодити з
      наявними шаблонами.
- [ ] Тести: 429 після N спроб; після вікна — знову можна; logout GET більше
      не розлогінює.

## Підводні камені

- In-memory ліміт скидається рестартом і не бачить другий процес — свідомий
  компроміс (бот сесій не має, веб — один процес); написати це в код-коментар,
  щоб ніхто не «чинив» багом.
- Ephemeral-ключ ламає «переживання рестарту» сесіями і **не має** зачепити
  шифрування креденшелів (там ключ обов'язковий — `crypto` вже падає без нього).
- Тести роутерів логіняться багато разів поспіль одним email — фікстурі
  треба або вимкнути ліміт, або підняти поріг, інакше сьют почервоніє.
