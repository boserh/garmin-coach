# OPS-06 · Метрики кешів: hit-rate і розміри на одній сторінці

**Тип:** спостережуваність (BA-аналіз 2026-07-23, кат. B) · **Оцінка:** S ·
**Пріоритет:** низький · **Поверхня:** веб (admin) · **Залежності:** ST-20 (та сама
сторінка `/admin/cache`; можна робити в одному PR)

## Проблема

PERF-02 économить реальні гроші, але ефективність невидима: cache-hit-share є лише в
`/costs` за місяць (одним числом), а по kind — ніде; розмір `GARMIN_CACHE_DIR` на
SD-картці і частка прострочених рядків `llm_cache` не видні ніде, крім ssh. Немає
способу помітити «кеш перестав хітити після зміни ключа» (README-пастка №1 цього
проєкту — саме такий баг уже був з volatile `generated`).

## User story

Як адмін я хочу бачити hit-rate дедуп-кешу по kind за 30 днів і розміри кешів, щоб
помічати регресії ключа кешу та контролювати місце на SD.

## Обсяг

**Входить:** read-only блоки на `/admin/cache` (сторінка зі ST-20): hit-rate по
`kind` за 7/30 дн (з `report_logs.cached` — дані вже є, один GROUP BY), рядки/байти
llm_cache (+частка прострочених), файли/байти garmin-кешу за префіксами, вік
найстарішого файлу.

**Не входить:** часові ряди/графіки (достатньо таблиці); експорт метрик
(Prometheus — інший масштаб).

## Acceptance criteria

- [ ] Hit-rate по kind 7/30 дн збігається з ручним підрахунком по `report_logs`
      (тест на фікстурах).
- [ ] Розміри рахуються за один прохід каталогу; сторінка рендериться <200мс на
      Pi-масштабі даних (без рекурсивного stat-шторму — каталог плаский).
- [ ] Нульові стани (порожній кеш) рендеряться без помилок.

## Технічні дотики

- `app/garmin/repository/stats.py`: `cache_hit_stats(session, days)` (GROUP BY kind,
  cached).
- `app/db/llm_cache.py::stats()`, `app/garmin/client.py::cache_stats()` (спільні зі
  ST-20).
- `app/routers/admin.py` + `cache.html`.
