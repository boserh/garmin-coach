# ST-11 · Ворнінг про смерть garth-токена в morning-тику

**Тип:** покращення / ops (BA-аналіз 2026-07-21) · **Оцінка:** S · **Пріоритет:**
високий (перетворює OPS-01-ризик з «раптово все впало» на планову дію) ·
**Залежності:** OPS-01 ✅ (`app/garmin/token_info.py`, CLI `token-expiry`)

## Проблема

OAuth1-токен живе ~1 рік від видачі; дедлайн кожного юзера вже вміє рахувати CLI
`token-expiry` — але його треба **згадати запустити**. Коли токен помре, ранкова
джоба почне падати в MFA/логін-фейли, і перелогін стане пожежею (особливо в
Cloudflare-еру, коли свіжий логін — не гарантія).

## Acceptance criteria

- [ ] Хук у morning-тику (`bot/jobs.py`, поруч з MFA-нотифікацією): декодувати
      дедлайн збереженого токена через наявний `token_info`; якщо лишилось ≤30
      або ≤7 днів — DM «токен Garmin спливає ~<дата>, перелогінься в /settings».
- [ ] Guard через `bot_state` (патерн `mfa_notified:<date>`): один DM на поріг
      (30д і 7д), не щоранку — ключ типу `token_warn:<threshold>` скидається
      після оновлення токена (нова дата видачі → нові ключі).
- [ ] Декодування токена не робить мережевих викликів і не ламає тик при
      битому/відсутньому токені (try/except → скіп, як інші best-effort хуки).
- [ ] Тести: DM на 30/7-порогах; без повторів; свіжий токен → тиша.

## Підводні камені

- `token_info` читає raw SQL заради напівмігрованих БД — у тику користувач уже
  завантажений ORM'ом; винести чисту функцію «token bytes → issue/expiry date»,
  щоб не тягнути raw-SQL-шлях у джобу.
- Дата видачі відома точно лише бо токен персистять одразу після свіжого логіну
  (нотатка OPS-01) — якщо це колись зміниться, оцінка стане нижньою межею;
  формулювати DM як «приблизно».
